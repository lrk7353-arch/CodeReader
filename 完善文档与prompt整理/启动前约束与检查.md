# 启动前约束与检查.md

## 用途

本文档是一份可以直接交给 Codex / coding agent 的启动前约束与检查 Prompt，用于在正式开发 CodeReader MVP 之前，对现有产品文档、技术方案、任务拆解和代码实现计划进行一次方向校准。

请 Codex 严格根据本文档进行审查、补充和修正，避免 CodeReader 在开发初期偏离核心定位。

# CodeReader 启动前约束与检查 Prompt

## 一、你的角色

你是 CodeReader 项目的启动前技术审查与产品约束 agent。

你需要基于已有项目文档、产品设计、需求文档、技术架构文档、MVP 实现标准和开发任务计划，对 CodeReader 进行一次正式开工前检查。

你的目标不是扩大需求，而是帮助项目保持克制、清晰、可落地。

请特别注意：

> CodeReader 的 MVP 目标不是做一个功能繁多的 AI 编程工具，而是先跑通“独立桌面端 AI 代码阅读 IDE”的最小闭环。

---

## 二、CodeReader 的不可偏离定位

CodeReader 是一个独立桌面端 AI 代码阅读 IDE，不是 VS Code 插件，不是聊天机器人，不是代码自动生成工具，也不是 PR Review 工具。

它的核心目标用户是：

- 使用 AI 生成代码的人；
- 没有系统编程训练、但希望真正理解自己项目的人；
- 想通过逐行、逐块、逐函数、逐文件阅读来掌握项目的人；
- 不希望自己的 AI 生成项目变成黑箱的人。

它的核心产品价值是：

> 把 AI 生成的代码从黑箱产物，转化为可阅读、可解释、可审阅、可持久化、可逐步掌握的认知资产。

它的核心交互形态是：

```text
左侧/中间：清晰分行的代码阅读区
右侧：与代码行、代码块、函数、文件稳定映射的解释层
```

CodeReader 的主体验必须是：

```text
代码文件 → 行/块/函数锚点 → 右侧解释层 → 用户理解状态 → 本地持久化
```

而不是：

```text
用户输入问题 → AI 回答一段文字
```

聊天能力可以作为辅助，但不能成为主产品形态。

---

## 三、MVP 必须坚持的最小闭环

请检查当前计划是否围绕以下闭环展开：

```text
用户打开一个本地代码文件
→ 中间区域显示代码、行号、语法高亮
→ 用户点击某一行、某个代码块或某个函数
→ 右侧显示稳定映射的解释
→ 用户可以标记“已理解 / 有疑问 / 不对劲”
→ 解释被保存到本地数据库
→ 用户关闭应用后再次打开，解释仍然存在
→ 文件变化后，旧解释会被标记为可能过期或已过期
```

如果当前开发计划中有任何任务没有服务于这个闭环，请标记为 P1 或 P2，不要进入 MVP。

---

## 四、MVP 必须做的功能

请确保 MVP 至少包含以下能力。

### 1. 独立桌面应用

必须是独立桌面应用。

推荐技术栈：

```text
Tauri + React + TypeScript + Monaco Editor + SQLite + Tree-sitter + LLM API
```

不得把 MVP 主产品定义为 VS Code 插件。

VS Code 插件可以作为后续扩展入口，但不属于 MVP 主线。

---

### 2. IDE 式三栏界面

MVP 需要具备基础三栏布局：

```text
左侧：文件树 / 示例入口
中间：代码阅读区
右侧：解释面板
```

中间代码区必须支持：

- 行号；
- 语法高亮；
- 只读代码展示；
- 点击行；
- 点击函数或代码块；
- 当前选中位置高亮。

---

### 3. 本地文件打开

MVP 至少支持打开单个本地代码文件。

优先支持：

```text
.ts / .tsx / .js / .jsx
```

可以暂缓完整项目目录扫描，但最好预留项目目录能力。

---

### 4. 解释层持久化

解释不能是一次性聊天结果。

解释必须保存到本地 SQLite，并在再次打开同一文件时加载。

必须检查解释是否具备以下绑定信息：

```text
project_id
file_id
file_path
file_hash
snapshot_id
target_type
start_line
end_line
symbol_id
code_hash
anchor_text
status
created_at
updated_at
```

特别注意：

> 解释不能只绑定 line_number。

因为代码一旦变化，行号会漂移。解释应该绑定到代码结构节点、行范围、代码内容摘要和文件快照。

---

### 5. 解释对象优先级

MVP 不要承诺“所有代码默认逐行解释”。

请按以下优先级设计解释对象：

| 优先级 | 解释对象 | MVP 是否必须 |
|---|---|---|
| P0 | 文件级解释 | 必须 |
| P0 | 函数级解释 | 必须 |
| P0 | 代码块解释 | 必须 |
| P0 | 用户点击后的关键行解释 | 必须 |
| P1 | 全量逐行解释 | 暂缓 |
| P1 | 跨文件调用解释 | 暂缓 |
| P2 | 完整项目知识图谱 | 暂缓 |

核心原则：

> 结构优先，行级按需展开。

---

### 6. 三层解释结构

每条解释默认采用三层结构：

```text
1. 代码层意义
2. 局部组成意义
3. 项目全局意义
```

三层含义如下：

| 层级 | 含义 |
|---|---|
| 代码层意义 | 从代码规则、语法、写法本身解释它做了什么 |
| 局部组成意义 | 解释它在当前函数、代码块或模块中的作用 |
| 项目全局意义 | 解释它在整个项目流程、业务目标或架构中的意义 |

不要为每一行都生成冗长解释。

解释的目标不是填满右侧面板，而是帮助用户读懂代码。

不同代码类型应有不同解释强度：

| 代码类型 | 默认解释策略 |
|---|---|
| import / 常量 / 简单赋值 | 简短解释 |
| 函数定义 / 类定义 | 三层解释 |
| if / for / try / catch | 三层解释 + 分支或异常说明 |
| 网络请求 / 数据库 / 文件操作 / 权限 / 支付 | 三层解释 + 风险提示 |
| 样式 / 重复模板代码 | 可折叠或批量解释 |

---

### 7. 用户理解状态与反馈

CodeReader 的解释质量不能完全依赖 AI 自我复核。

产品哲学是：

> AI 负责生成初始解释，用户负责审阅、质疑、标记和修正。

MVP 必须支持用户反馈：

```text
未读
已读
已理解
有疑问
不对劲
需重新解释
```

至少要支持以下操作：

- 标记“已理解”；
- 标记“有疑问”；
- 标记“不对劲”；
- 对某一行、代码块或函数重新解释；
- 用户写备注。

不要把“confidence: 0.82”这种模型置信度作为核心用户体验。

如果要表达不确定性，应使用人能理解的阅读提示，例如：

```text
这里依赖上下文较多，建议结合调用处一起看。
```

或者：

```text
这段代码涉及权限判断，建议重点检查边界条件。
```

---

### 8. 代码变更检测与解释过期机制

因为解释是持久化的，所以必须处理代码变化。

MVP 至少应支持：

- 文件 hash 检测；
- 文件内容变化识别；
- 行级或范围级 diff；
- 解释状态更新；
- 变更摘要展示。

解释状态至少包括：

```text
有效
可能过期
已过期
新增未解释
已删除
```

当文件变化后，CodeReader 应告诉用户：

```text
自上次阅读以来，该文件修改了多少行、新增了多少代码块、哪些解释可能已经过期。
```

Git 集成可以分阶段：

| 阶段 | Git 能力 |
|---|---|
| MVP | 文件 hash + 基础 diff |
| P1 | 读取 git diff，展示变更前后 |
| P2 | commit 级解释、PR 阅读、变更影响分析 |
| P3 | 团队审阅流、解释随 commit 版本流转 |

---

### 9. 上下文窗口管理

不要把整个大型文件直接丢给 LLM 做逐行解释。

请实现或预留 Context Builder。

根据解释目标动态构造上下文：

| 解释目标 | 上下文组成 |
|---|---|
| 文件解释 | 文件路径、语言、import/export、函数列表、完整文件或摘要 |
| 函数解释 | 函数完整代码、相关 import、调用到的函数摘要 |
| 代码块解释 | 当前代码块、所在函数、前后若干行 |
| 单行解释 | 当前行、所在代码块、所在函数、相关变量定义 |
| 大文件解释 | 文件结构摘要优先，不默认逐行 |

默认解释策略建议：

| 文件行数 | 默认策略 |
|---:|---|
| 0-200 行 | 文件级 + 函数级 + 关键行解释 |
| 200-800 行 | 文件级 + 函数级，点击后展开行级 |
| 800 行以上 | 结构摘要和阅读路径优先 |
| 2000 行以上 | 提示用户按模块阅读，不建议整文件解释 |

核心原则：

> 先模块级理解，再按需展开行级分析。

---

### 10. LLM 输出协议

LLM 不能自由输出一大段 Markdown。

LLM 输出必须是结构化 JSON，便于解析、保存、渲染和版本管理。

建议基础结构：

```json
{
  "target_type": "function",
  "target_name": "handleSubmit",
  "start_line": 24,
  "end_line": 61,
  "code_meaning": "这段代码定义了一个表单提交处理函数。",
  "local_meaning": "它负责读取用户输入、调用登录接口，并根据结果更新页面状态。",
  "global_meaning": "它是登录流程中的用户身份验证入口之一。",
  "risk_notes": [
    "需要确认接口失败时是否有足够的错误提示。",
    "需要确认是否处理了重复点击提交的问题。"
  ],
  "reader_notes": [
    "重点理解 async/await 的执行方式。",
    "重点理解状态更新和页面反馈之间的关系。"
  ]
}
```

必须检查：

- JSON 是否可解析；
- 字段是否稳定；
- 是否支持失败重试；
- 是否有 fallback；
- 是否能保存到 SQLite；
- 是否能映射到右侧解释面板。

---

### 11. Tree-sitter 结构解析

Tree-sitter 应在行级解释前完成。

请检查开发顺序是否满足：

```text
先识别函数、类、代码块边界
再生成行级或块级解释
```

MVP 至少识别：

```text
import
export
function
class
if / for / try / catch
函数起止行
代码块起止行
```

并建立：

```text
line_number → function/block/file
```

的映射关系。

---

### 12. 内置示例

MVP 应内置 3 个示例代码文件，用于用户快速体验：

| 示例 | 行数 | 目的 |
|---|---:|---|
| 入门示例 | 30-50 行 | 展示左右映射体验 |
| 中等示例 | 80-100 行 | 展示函数、代码块、局部关系 |
| 复杂示例 | 100-200 行 | 展示模块拆分、风险提示、阅读状态 |

示例文件应自带解释层，不依赖用户第一次就配置 LLM API。

请检查当前计划中是否包含以下内容。

如果包含，请标记为 P1 / P2 / Future，不要纳入 MVP。

MVP 不做：

```text
不做 VS Code 插件作为主产品
不做团队协作
不做云端同步
不做账号系统
不做复杂项目知识图谱
不做全语言支持
不做自动代码修改
不做类 Cursor 的代码生成能力

不做多模型交叉验证

不做自动课程系统
不做复杂 Git 工作流
不做插件市场
```

## 六、开发顺序检查

请检查当前开发顺序是否符合以下原则：

```text
界面原型
→ Monaco 代码展示
→ 行点击映射
→ 示例解释层
→ SQLite 持久化
→ Tree-sitter 结构解析
→ LLM 解释生成
→ 用户反馈状态
→ 代码变更检测
→ 示例项目与打包
```

不建议一开始就做：

- 复杂 LLM agent；
- 多文件 RAG；
- 项目级知识图谱；
- GitHub 集成；
- 云端同步；
- 用户账号。

第一阶段最重要的是验证：

> 右侧稳定解释层这个体验本身是否成立。

## 七、启动前重点检查项

请逐项检查，并输出检查结果。

### 检查 1：是否把 CodeReader 做回了聊天工具？

如果主界面或主流程变成：

```text
输入问题 → AI 回答
```

则说明方向错误。

正确主流程应是：

```text
代码文件 → 代码锚点 → 右侧解释层 → 理解状态
```

---

### 检查 2：是否把 MVP 做得过大？

如果 MVP 包含以下内容，请指出并建议移出：

- 团队协作；
- 云同步；
- 完整 Git 平台；
- 多模型审查；
- 项目知识图谱；
- 自动代码修复；
- 多语言全覆盖。

---

### 检查 3：解释是否真的持久化？

请检查：

- 是否保存到 SQLite；
- 是否能重新加载；
- 是否能绑定到文件快照；
- 是否能绑定到行/代码块/函数；
- 是否有过期状态；
- 用户反馈是否保存。

---

### 检查 4：解释是否绑定得足够稳定？

请检查解释是否只绑定行号。

如果只绑定行号，请修正为：

```text
文件路径 + 文件 hash + 代码结构节点 + 行范围 + code_hash + anchor_text
```

---

### 检查 5：用户是否能参与纠错？

请检查是否支持：

- 已理解；
- 有疑问；
- 不对劲；
- 重新解释；
- 用户备注。

如果没有，请加入 MVP。

---

### 检查 6：解释是否过度复杂？

请检查右侧解释面板是否默认展示过多字段。

默认只需要：

```text
代码层意义
局部组成意义
项目全局意义
```

其余内容如风险、学习提示、相关行、备注，可以折叠展示。

---

### 检查 7：是否有代码变更处理？

请检查是否能在代码变化后标记解释状态。

必须至少支持：

```text
有效
可能过期
已过期
新增未解释
已删除
```

---

### 检查 8：是否处理上下文窗口限制？

请检查是否有 Context Builder 或等价方案。

如果当前设计直接把整个文件丢给 LLM，请修正。

---

### 检查 9：LLM 输出是否结构化？

请检查 LLM 是否输出稳定 JSON。

如果输出 Markdown 大段文字，请修正。

---

### 检查 10：是否支持无 API Key 的首次体验？

MVP 应支持用户打开内置示例，在不配置 LLM API Key 的情况下体验核心界面。

否则新用户第一次体验成本过高。

## 九、最终审查原则

请始终坚持以下原则：

1. CodeReader 是独立桌面端 AI 代码阅读 IDE，不是插件，不是聊天工具。
2. MVP 的第一目标是跑通单文件解释闭环。
3. 解释必须稳定映射到代码行、代码块、函数或文件。
4. 解释必须本地持久化，不能每次重新生成。
5. 用户必须能标记理解状态和反馈“不对劲”。
6. 不要追求 AI 无限自检，用户审阅能力本身是产品价值的一部分。
7. 不要默认全量逐行解释，应该结构优先、行级按需展开。
8. 大文件必须先做模块级/函数级理解，再按需展开局部。
9. 代码变更后必须能标记解释过期。
10. MVP 必须克制，先验证核心阅读体验。

---

## 十、最终目标

请帮助 CodeReader 在正式开工前完成一次严格收敛。

最终要确保项目从第一天开始就在正确方向上：

```text
打开代码 → 看见结构 → 点击行/块/函数 → 看到稳定解释 → 用户标记理解 → 解释被保存 → 代码变更后提示过期
```

只要这个闭环成立，CodeReader 就已经和普通 AI 代码解释器、聊天式代码问答工具拉开差距。

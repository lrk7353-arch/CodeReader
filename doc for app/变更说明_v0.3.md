# CodeReader 文档变更说明 v0.3

## 1. 变更原因

v0.2 已经将 CodeReader 的主线从 VS Code 插件修正为独立桌面端 AI 代码阅读 IDE。v0.3 进一步补充产品和工程闭环中容易被低估的部分：

- 解释质量不能只依赖 AI 自评；
- 非科班用户需要白话化、分层化、可展开的解释方式；
- 用户第一次打开陌生项目时需要阅读路径，而不是只看到文件树；
- 持久化解释层必须面对代码频繁变更；
- LLM 上下文必须由 Context Builder 管理；
- 项目级认知地图需要先有轻量存储结构；
- 开发顺序应先做结构解析，再做行级解释生成。

## 2. 核心增强

| 方向 | v0.2 状态 | v0.3 增强 |
|---|---|---|
| 解释质量 | 主要依靠结构化输出和置信度 | 加入用户反馈、重新解释、解释版本和可读可信提示 |
| 解释结构 | 字段较多，容易信息过载 | 收敛为三层核心解释，并支持白话/详细切换 |
| 新手体验 | 以功能列表为主 | 加入示例项目、推荐阅读路径和阅读进度 |
| 代码变更 | 标记解释过期 | 增加变更摘要视图和局部重新解释 |
| 上下文管理 | 只提到 Context Builder | 明确不同解释目标的上下文构造策略 |
| 数据模型 | 以文件、代码节点、解释节点为主 | 增加 feedback、change_records、project_maps、reading_paths |
| 开发顺序 | 较早进入行级解释 | 调整为先 Tree-sitter 结构锚定，再做行级解释 |

## 3. 新增产品判断

CodeReader 不应陷入“AI 解释代码，再让 AI 检查解释”的循环。更合理的闭环是：

```text
AI 生成初始解释
  ↓
用户阅读、标记、质疑
  ↓
系统保存反馈与阅读状态
  ↓
必要时局部重新解释
  ↓
解释层随代码和用户理解持续演进
```

## 4. 新增开发判断

MVP 开发顺序应遵循：

```text
先验证阅读体验
→ 再建立结构锚点
→ 再做持久化解释层
→ 再接入 LLM
→ 最后做变更摘要和项目级阅读路径
```

这能避免在缺少代码结构锚点时过早生成大量脆弱的行级解释。

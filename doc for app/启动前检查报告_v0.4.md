# CodeReader 启动前检查报告

版本：v0.4  
检查范围：MVP 级别实现收束  
检查结论：本检查只约束 MVP，不削弱 CodeReader 的全量设定与长期目标。

---

## 1. 总体判断

- 是否可以启动：可以。
- 最大风险：MVP 在实现中滑向“聊天式代码问答”或“完整项目知识图谱”，导致单文件解释闭环没有先跑通。
- 最需要先修正的问题：解释锚点必须稳定，用户理解状态必须进入 P0，首次体验必须不依赖 API Key。

本轮已补充或修正：

- 将“已理解状态标记”统一为 P0；
- 将“行级解释”收束为“用户点击后的关键行按需解释”；
- 明确 MVP 项目地图只是轻量文件地图和阅读路径，不是完整知识图谱；
- 补齐解释目标所需的 `snapshot_id`、`file_hash`、`symbol_id`、`anchor_text` 等稳定锚点字段；
- 明确内置示例和预置解释层必须支持无 API Key 首次体验；
- 将聊天追问移出 MVP 默认入口。

---

## 2. MVP 边界检查

### 应保留在 MVP 的内容

- 独立桌面端应用；
- IDE 式三栏界面；
- 打开本地单文件和小型项目；
- Monaco 只读代码展示、行号、语法高亮、单行点击、多行选择；
- Tree-sitter 识别函数、类、代码块和 import/export；
- 文件级、函数级、代码块级、关键行按需解释；
- SQLite 本地持久化；
- 用户阅读状态和解释反馈；
- 代码变更检测、过期标记和变更摘要；
- 内置示例和预置解释层；
- 轻量项目文件地图和推荐阅读路径。

### 应移出 MVP 的内容

- VS Code 插件主产品；
- 团队协作；
- 云同步；
- 完整 Git 平台；
- 多模型交叉审查；
- 完整项目知识图谱；
- 自动代码修复；
- 多语言全覆盖；
- PR Review 平台；
- 聊天追问默认主入口。

这些能力可以保留在 P1/P2/Future 中，不能进入 MVP 完成定义。

---

## 3. 产品形态检查

- 是否仍是独立桌面端 AI 代码阅读 IDE：是。
- 是否出现聊天工具化倾向：主文档中没有主线偏移；“聊天追问”已被明确为 P1，MVP 不做默认入口。
- 修正建议：开发时必须坚持“代码锚点 -> 右侧解释层 -> 用户理解状态”的主流程，不能优先实现自由问答窗口。

---

## 4. 技术架构检查

- Tauri / React / Monaco 是否合理：合理，匹配独立桌面端和 IDE 式阅读体验。
- SQLite 持久化是否完整：基本完整，已补充解释目标稳定锚点字段。
- Tree-sitter 解析是否前置：已前置到行级解释之前，符合“结构优先，行级按需展开”原则。
- Context Builder 是否明确：已明确不同解释目标的上下文构造策略。
- LLM JSON 协议是否明确：已明确结构化 JSON、校验失败不写入数据库、失败重试和 fallback。

---

## 5. 数据模型检查

- 当前 schema 缺少的表：MVP 必需表已覆盖；完整跨文件关系图可后续扩展。
- 当前 schema 缺少的字段：已补充 `snapshot_id`、`file_path`、`file_hash`、`symbol_id`、`anchor_text` 和三层解释字段。
- 解释锚点是否稳定：修正后稳定，不再只依赖行号。
- 修正建议：实现时将 `explanation_targets` 作为解释绑定的核心表，避免 UI 或业务逻辑直接按裸行号查解释。

MVP 解释目标至少应保存：

```text
project_id
file_id
file_path
file_hash
snapshot_id
target_type
start_line
end_line
symbol_id
code_hash
anchor_text
status
created_at
updated_at
```

---

## 6. 开发顺序检查

- 当前顺序是否合理：总体合理。
- 是否需要调整：不需要大改，但必须把“无 API Key 示例体验”和“mock 解释持久化”作为早期验收点。
- 推荐顺序：

1. 纯网页体验原型：Monaco + 预置解释 JSON + 反馈按钮；
2. Tauri + React 基础应用；
3. Monaco 只读代码展示、行点击、多行选择；
4. 本地文件/项目打开；
5. Tree-sitter 结构解析；
6. SQLite schema；
7. mock 解释保存和加载；
8. 单行、多行、函数解释面板；
9. Context Builder；
10. LLM 结构化解释生成；
11. 用户反馈、阅读状态和局部重新解释；
12. 变更检测、过期标记和变更摘要；
13. 轻量阅读路径、项目文件地图和内置示例；
14. MVP 打包。

---

## 7. MVP 验收清单

1. 独立桌面端应用可以启动；
2. 首次启动可进入内置示例，且不需要 API Key；
3. 可以打开单个 JS/TS 文件；
4. 可以打开小型本地项目并展示文件树；
5. 中间代码区显示行号、语法高亮和只读代码；
6. 单行点击能高亮并切换右侧解释；
7. 多行选择能映射到代码块解释；
8. Tree-sitter 能识别函数、类、代码块和 import/export；
9. 右侧默认展示三层解释；
10. 风险、前置知识、相关行和用户笔记可折叠展示；
11. 解释保存到 SQLite，再次打开可恢复；
12. 用户能标记已理解、有疑问、不对劲并写备注；
13. 代码变化后能显示 valid/stale/invalid/new_unexplained/deleted 状态；
14. 能展示变更摘要并只重新解释受影响区域；
15. 用户能用 CodeReader 完整读完一个关键文件。

---

## 8. 下一步执行任务

### 任务 1：体验原型

- 目标：验证右侧稳定解释层是否成立。
- 输入：3 个内置示例文件、预置解释 JSON。
- 输出：纯网页 Demo。
- 验收标准：无 API Key 可点击单行、多行、函数并展示对应解释。

### 任务 2：桌面端骨架

- 目标：搭建 Tauri + React + TypeScript 三栏应用。
- 输入：技术架构文档、MVP 实现标准。
- 输出：可运行桌面应用。
- 验收标准：左侧、代码区、右侧面板布局稳定。

### 任务 3：Monaco 阅读区

- 目标：实现 IDE 式只读代码阅读体验。
- 输入：示例代码、本地文件内容。
- 输出：行号、语法高亮、单行点击、多行选择。
- 验收标准：选择状态能同步到右侧解释面板。

### 任务 4：结构解析与锚点

- 目标：建立行/块/函数到结构节点的映射。
- 输入：JS/TS 文件。
- 输出：code_nodes、explanation_targets。
- 验收标准：解释不只绑定行号，具备快照、结构节点、代码 hash 和锚点文本。

### 任务 5：SQLite 持久化

- 目标：保存解释、阅读状态、反馈和变更记录。
- 输入：schema、mock 解释数据。
- 输出：本地数据库。
- 验收标准：关闭重开后解释和状态恢复，不自动重新调用 LLM。

### 任务 6：解释面板

- 目标：实现三层解释和反馈入口。
- 输入：解释 JSON、用户选择目标。
- 输出：右侧解释面板。
- 验收标准：默认三层展示，辅助信息折叠，反馈可保存。

### 任务 7：Context Builder 与 LLM

- 目标：按解释目标构造上下文并生成结构化解释。
- 输入：文件结构、代码节点、用户选择目标。
- 输出：可校验 JSON 解释。
- 验收标准：JSON 校验失败不入库，支持局部重新解释。

### 任务 8：变更检测

- 目标：处理代码变化后的解释有效性。
- 输入：旧快照、新文件内容、旧解释目标。
- 输出：过期状态和变更摘要。
- 验收标准：新增、修改、删除和受影响解释可被识别。

### 任务 9：第一公里引导与打包

- 目标：让新用户快速进入阅读。
- 输入：内置示例、轻量文件地图、推荐阅读路径。
- 输出：MVP 可安装包。
- 验收标准：用户无需 API Key 即可体验示例；真实文件可完成单文件解释闭环。

# CodeReader MVP 实现标准

版本：v0.4  
MVP 定位：独立桌面端 AI 代码阅读 IDE  
重要修正：VS Code 插件不是 MVP 主产品形态。

本文件只约束 MVP 级别实现，不削弱 CodeReader 的全量目标。项目级认知地图、跨文件关系、插件入口、协作和云同步可以保留在后续路线中，但不能成为 MVP 完成前置条件。

---

## 1. MVP 目标

CodeReader MVP 的目标是验证一个核心判断：

> 用户是否需要一个独立的、IDE 形态的工具，用来稳定、系统、逐行地理解 AI 生成代码。

MVP 不追求成为完整 IDE，也不追求支持所有语言，而是要把以下体验打磨清楚：

1. 用户可以打开项目；
2. 用户可以像在 IDE 中一样浏览代码；
3. 用户可以为代码生成解释层；
4. 解释层稳定绑定到代码行/块/函数/文件；
5. 解释层可以持久化保存；
6. 用户再次打开时不需要重新问 AI；
7. 用户可以标记理解状态、风险状态和解释反馈；
8. 用户面对陌生项目时知道从哪里开始读；
9. 代码变更后，用户知道发生了什么变化、哪些解释受影响。

---

## 2. MVP 产品形态

### 2.1 正确形态

MVP 应是：

> 一个可以下载安装的独立桌面端应用。

界面应类似轻量 IDE：

```text
文件树 + 代码阅读区 + 右侧解释面板
```

### 2.2 不正确形态

MVP 不应是：

- VS Code 插件；
- 浏览器里的普通网页解释器；
- 聊天机器人；
- 代码审查插件；
- 单次 prompt 工具；
- 代码注释生成器。

---

## 3. MVP 范围

### 3.1 必须实现的 P0 能力

| 模块 | 能力 | 验收标准 |
|---|---|---|
| 桌面端 | 独立应用启动 | 用户可安装/运行应用 |
| 项目打开 | 打开本地文件夹 | 文件树正确展示 |
| 示例体验 | 内置示例项目 | 首次启动可直接体验 CodeReader 的行/块/函数解释映射 |
| 无 API Key 体验 | 预置解释层可用 | 用户不配置 API Key 也能体验核心阅读界面 |
| 文件打开 | 打开代码文件 | 中间代码区展示内容 |
| 代码阅读 | 行号和语法高亮 | 用户能清晰逐行阅读 |
| 行级选择 | 点击某一行 | 当前行高亮，右侧切换解释 |
| 多行选择 | 选择代码块并查看块级解释 | 多行选择高亮，右侧展示对应代码块解释 |
| 结构锚定 | Tree-sitter 识别函数/代码块 | 行级和多行解释能绑定到结构节点 |
| 上下文构造 | Context Builder 构造解释上下文 | 不无脑发送整文件，按目标选择必要上下文 |
| 解释生成 | 调用 LLM 生成解释 | 至少支持文件级、函数级、代码块级、关键行级解释 |
| 解释结构 | 固定字段输出 | 以代码层意义、局部组成意义、项目全局意义为核心 |
| 解释持久化 | 保存到本地数据库 | 再次打开无需重新生成 |
| 解释加载 | 根据当前行读取解释 | 点击行即时显示 |
| 过期检测 | 文件变化后标记解释过期 | 用户能看到解释状态变化 |
| 变更摘要 | 展示代码变化概览 | 显示新增、修改、删除及受影响解释 |
| 局部重新解释 | 重新解释当前行/块 | 不需要整文件重新生成 |
| 推荐阅读路径 | 推荐先读哪些文件 | 打开项目后能看到入口、配置、核心文件建议 |
| 阅读状态 | 标记已理解/有疑问/有风险 | 状态可保存和恢复 |
| 解释反馈 | 标记有帮助/不对劲 | 反馈保存到本地并可触发重新解释 |

### 3.2 P1 能力

| 模块 | 能力 |
|---|---|
| 函数导航 | 点击函数名查看函数解释 |
| 关系高亮 | 高亮依赖行和影响行 |
| 风险聚合 | 当前文件风险列表 |
| 搜索 | 搜索解释内容和代码符号 |
| 导出 | 导出解释层为 Markdown/JSON |
| Git diff | 使用 Git 提供更精确变更来源 |

### 3.3 暂不实现的 P2 能力

- VS Code 插件；
- JetBrains 插件；
- 团队协作；
- 云同步；
- GitHub PR 审查；
- 自动修复代码；
- 完整架构图；
- 多模型路由；
- 本地大模型支持；
- 所有语言支持。

## 4. MVP 支持语言

### 4.1 推荐首选

MVP 首选支持：

- JavaScript；
- TypeScript；
- TSX/JSX 可作为扩展。

原因：

- AI 生成应用中前端/Node 项目常见；
- Tree-sitter 支持成熟；
- 文件结构适合做模块和函数解释；
- 用户群体与独立开发者、AI builder 高度重合。

### 4.2 第二阶段支持

- Python；
- Markdown 配置/说明文件；
- JSON/YAML 配置文件。

## 5. MVP 核心页面

### 5.1 启动页

功能：

- 示例项目体验区；
- 打开项目；
- 打开单文件；
- 最近项目列表；
- API Key/模型配置入口。

### 5.2 主阅读页

布局：

```text
左侧：文件树
中间：代码阅读区
右侧：解释面板
底部：状态栏
```

主阅读页还应包含：

- 当前文件解释状态；
- 阅读进度；
- 变更摘要入口；
- 推荐阅读路径入口；
- 解释反馈入口。

### 5.3 设置页

功能：

- LLM Provider 设置；
- API Key 设置；
- 默认模型；
- 忽略目录；
- 数据存储位置；
- 隐私设置。

## 6. MVP 解释字段标准

### 6.1 行级解释

每一行解释至少包含：

| 字段 | 必须 | 说明 |
|---|---|---|
| code | 是 | 当前行代码 |
| code_level_meaning | 是 | 代码层意义：这行代码写了什么 |
| local_composition_meaning | 是 | 局部组成意义：它在当前函数/代码块中干什么 |
| project_role_meaning | 是 | 项目全局意义：它在项目流程中的位置 |
| risks | 是 | 潜在风险 |
| prior_knowledge | 是 | 前置知识/学习提示 |
| review_suggestion | 是 | 审阅建议 |
| depends_on_lines | 否 | 依赖行 |
| affects_lines | 否 | 影响行 |
| trust_label | 是 | 面向用户的可信提示 |
| trust_reason | 是 | 为什么这样提示 |
| display_mode | 是 | plain 或 detailed |

旧字段 `surface_meaning`、`actual_meaning`、`relational_meaning` 可以作为兼容字段保留，但 MVP UI 默认展示三层解释。

### 6.2 函数级解释

至少包含：

- 函数职责；
- 参数说明；
- 返回值；
- 副作用；
- 调用关系；
- 失败场景；
- 关键行说明；
- 风险点。

### 6.3 文件级解释

至少包含：

- 文件职责；
- 主要导入依赖；
- 主要导出内容；
- 核心函数/类；
- 执行流程摘要；
- 在项目中的位置；
- 文件级风险。

### 6.4 解释反馈

每条解释至少支持：

- 有帮助；
- 不对劲；
- 更白话；
- 更详细；
- 重新解释；
- 用户笔记。

右侧解释面板默认只展示“代码层意义、局部组成意义、项目全局意义”三层核心解释。风险、前置知识、审阅建议、相关行、可信提示和用户笔记作为可展开信息展示，避免 MVP 第一版变成字段堆叠。

---

## 7. MVP 数据持久化标准

MVP 必须本地保存：

- 项目信息；
- 文件信息；
- 文件 hash；
- 代码节点；
- 解释节点；
- 解释目标；
- 解释反馈；
- 用户阅读状态；
- 代码快照；
- 变更记录；
- 项目文件地图；
- 推荐阅读路径；
- 模型信息；
- prompt/schema 版本。

### 7.1 再次打开标准

用户关闭 CodeReader 后再次打开同一项目：

- 文件树应恢复；
- 已解释文件应显示“已解释”；
- 点击已解释行，应直接展示解释；
- 不应自动重新调用 LLM；
- 如果文件变化，应显示过期状态。

---

## 8. MVP 解释稳定性标准

| 情况 | 期望行为 |
|---|---|
| 文件未变化 | 加载旧解释 |
| 文件局部变化 | 标记相关解释过期 |
| 行号变化但代码未变 | 尽量迁移解释 |
| 用户手动重新解释 | 生成新解释版本 |
| LLM 返回失败 | 保留旧解释并提示错误 |
| 解释格式不合法 | 不写入数据库，提示重试 |
| 用户标记“不对劲” | 保留旧解释并允许局部重新解释 |
| 新增代码 | 标记为新增未解释 |
| 原代码删除 | 原解释标记为已删除或归档 |

## 9. MVP 体验验收用例

### 用例 1：首次打开项目

1. 用户启动 CodeReader；
2. 看到“体验示例项目”和“打开项目”入口；
3. 点击“打开项目”；
4. 选择一个 JS/TS 项目；
5. 左侧展示文件树；
6. 系统展示推荐阅读路径；
7. 用户点击一个 `.ts` 文件；
8. 中间显示代码。

验收：通过。

### 用例 2：生成解释层

1. 用户点击“分析当前文件”；
2. 系统显示分析进度；
3. 分析完成后，文件状态变为“已解释”；
4. 用户点击某一行；
5. 右侧出现结构化解释。

验收：通过。

### 用例 3：持久化解释

1. 用户关闭应用；
2. 重新打开 CodeReader；
3. 打开同一项目和文件；
4. 点击已解释行；
5. 右侧直接显示上次解释。

验收：通过。

### 用例 4：代码变更

1. 用户修改一个已解释文件；
2. 返回 CodeReader；
3. 系统检测文件变化；
4. 系统展示变更摘要；
5. 受影响解释标记为“已过期”；
6. 新增代码标记为“新增未解释”；
7. 用户可以只重新生成受影响区域。

验收：通过。

### 用例 5：阅读状态

1. 用户点击某一行；
2. 阅读解释；
3. 标记“已理解”；
4. 关闭并重新打开；
5. 状态仍然存在。

验收：通过。

### 用例 6：解释反馈

1. 用户点击某一行；
2. 右侧展示解释；
3. 用户标记“有帮助”或“不对劲”；
4. 对“不对劲”的解释选择“用更白话解释”或“重新解释”；
5. 新解释作为新版本保存。

验收：通过。

## 10. MVP 完成定义

MVP 可以被认为完成，当满足：

1. 独立桌面端可运行；
2. 能打开本地项目；
3. 能展示文件树和代码；
4. 能生成并保存解释层；
5. 能行级映射解释；
6. 能再次打开直接加载解释；
7. 能标记解释过期；
8. 能展示变更摘要并局部重新解释；
9. 能记录用户阅读状态和解释反馈；
10. 能生成推荐阅读路径；
11. 能提供内置示例体验；
12. 用户首次启动不配置 API Key，也能通过内置示例理解行/块/函数到解释层的映射；
13. 用户可以用它完整读懂一个关键代码文件。

---

## 11. MVP 开发顺序

1. 纯网页体验原型：Monaco + 预置解释 JSON + 反馈按钮；
2. Tauri + React 基础应用；
3. Monaco Editor 集成；
4. 本地文件/项目打开；
5. 文件树和代码展示；
6. Tree-sitter 结构解析，识别函数、代码块、import/export；
7. SQLite 接入；
8. 解释层 schema、target、feedback、reading_state、change_record；
9. mock 解释保存和加载；
10. 单行、多行、函数解释面板；
11. Context Builder；
12. LLM 解释生成；
13. 解释反馈、白话/详细模式、局部重新解释；
14. 变更检测、过期标记、变更摘要；
15. 推荐阅读路径、项目文件地图、阅读进度；
16. 内置示例与打包。

---

## 12. MVP 后第一轮迭代建议

MVP 后优先迭代：

1. 解释层导出；
2. 代码块和函数关系图；
3. 多条可选阅读路径，例如“快速理解”“风险优先”“学习优先”；
4. 跨文件依赖关系；
5. 在 Python、SQL 基础上继续扩展高需求语言；
6. 风险集中面板；
7. 从 VS Code 打开 CodeReader 的轻量插件。

## 13. MVP 发行验收结果

截至 2026 年 6 月 11 日，Task 12 已将上述 MVP 闭环收束为 Windows x64 可安装桌面应用：

1. NSIS `Setup.exe` 与 MSI 均由 Tauri 正式 release 构建生成，不以 debug 可执行文件充当交付物；
2. 安装后可在没有 Vite、Node 开发服务器和独立后端进程的情况下启动；
3. 内置示例、真实项目树、推荐阅读路径、代码预览、结构解释、阅读状态、反馈、持久化和变更检测均保留；
4. JavaScript/TypeScript、Python、SQL 纳入当前结构化解释范围，常见非支持文本按能力只读预览；
5. 重启可恢复本地阅读状态，卸载默认保留用户数据库与系统凭据，重新安装后可继续恢复；
6. 安装产物不提交 Git，只提交可复现的配置、图标、脚本和验收文档。

这是一条 MVP 可交付基线，不代表全量产品完成。完整项目认知地图、更广语言覆盖、团队协作、云同步、插件入口等长期设定继续保留。

# CodeReader 项目文档包

版本：v0.4  
更新重点：在“独立桌面端 AI 代码阅读 IDE”路线基础上，完成启动前 MVP 收束检查，明确哪些约束只作用于 MVP 实现，不削弱全量产品目标。

## 1. 项目一句话定义

**CodeReader 是一个独立的、IDE 形态的 AI 代码阅读器，通过持久化的行级解释层，帮助 AI 编程用户逐行、逐块、逐函数、逐文件理解并掌握自己的代码项目。**

它不是“输入一段代码，AI 回复一段解释”的问答工具，也不是依附在现有 IDE 里的普通插件，而是一个围绕“代码认知权”重新设计的代码阅读环境。

## 2. 核心定位

CodeReader 面向的是正在使用 AI 生成代码、但缺乏系统编程训练的用户。它解决的问题不是“AI 能不能帮我写代码”，而是：

> AI 可以替我生成代码，但我必须能够理解、审阅并掌握这些代码，否则项目对我而言仍然是黑箱。

CodeReader 的核心目标是把 AI 生成代码从黑箱产物转化为：

- 可逐行阅读的代码文本；
- 可稳定保存的解释层；
- 可追溯的行间关系；
- 可审阅的风险提示；
- 可沉淀的项目认知资产。

## 3. 产品形态原则

| 原则 | 说明 |
|---|---|
| 独立产品优先 | MVP 是独立桌面端应用，不以 VS Code 插件作为主产品形态。 |
| IDE 式界面 | 采用类似 VS Code / PyCharm 的文件树、代码区、解释面板布局。 |
| 代码是主体 | 用户面对的是代码文件，而不是聊天窗口。 |
| 行级映射 | 解释信息稳定绑定到代码行、代码块、函数、文件等结构节点。 |
| 持久化解释 | 同一文件再次打开时解释信息仍然存在，不随机变化、不重复生成。 |
| 增量更新 | 代码变更后只重新分析受影响的解释节点。 |
| 人参与校正 | AI 生成初始解释，用户通过“有帮助/不对劲/重新解释/补充笔记”修正解释层。 |
| 第一公里引导 | 用户打开陌生项目后，应看到推荐阅读路径、示例入口和阅读进度，而不是只面对文件树。 |
| 上下文优先 | 解释生成必须由 Context Builder 选择必要上下文，不把整项目无脑塞给模型。 |
| 插件为补充 | VS Code 插件可作为后续导入入口或分发渠道，不是 MVP 的核心定义。 |

## 5. 推荐技术主线

MVP 推荐采用：

```text
Tauri + React + Monaco Editor + SQLite + Tree-sitter + LLM API
```

其中：

- **Tauri**：提供独立桌面端应用外壳；
- **React**：构建文件树、代码区、解释面板等界面；
- **Monaco Editor**：承载类似 IDE 的代码展示与行级交互；
- **SQLite**：本地持久化解释层、文件索引、阅读状态；
- **Tree-sitter**：进行语法树解析、函数/代码块识别、增量变更定位；
- **LLM API**：生成代码层意义、局部组成意义、项目全局意义、风险提示等解释内容。

## 6. v0.3 增强重点

v0.3 明确 CodeReader 的解释体验不是“AI 给出一个置信度数字”，而是一个可被用户理解、质疑、修正和积累的阅读闭环。

| 增强项 | 说明 |
|---|---|
| 三层解释结构 | 以“代码层意义、局部组成意义、项目全局意义”为核心，避免每行解释字段过载。 |
| 解释质量反馈 | MVP 中提供“有帮助/不对劲/重新解释/补充笔记”的简单入口，反馈用于后续 prompt 优化。 |
| 白话/详细模式 | 默认使用白话解释，必要时展开技术细节，降低非科班用户第一次阅读门槛。 |
| 示例项目体验区 | 内置 3 个示例代码，让用户先体验 CodeReader 的左右映射和解释层价值。 |
| 推荐阅读路径 | 打开项目后给出建议先读哪些文件，帮助用户跨过“面对文件树不知道从哪里开始”的第一公里。 |
| 变更摘要视图 | 代码变化后不只显示过期，而是说明新增、修改、删除及受影响解释。 |
| Context Builder | 为文件、函数、代码块、行解释分别选择上下文，控制 token 成本并减少误解。 |

## 7. v0.4 启动前收束重点

v0.4 不改变 CodeReader 的长期目标，只约束 MVP 的第一版实现。MVP 必须先跑通以下闭环：

```text
打开代码 -> 看见结构 -> 点击行/块/函数 -> 看到稳定解释 -> 用户标记理解 -> 解释被保存 -> 代码变更后提示过期
```

| 收束项 | MVP 口径 |
|---|---|
| 单文件闭环优先 | 先确保一个 JS/TS 文件可以完成解释、保存、恢复和过期检测。 |
| 结构优先 | 先识别函数、代码块和文件结构，再做行级按需解释。 |
| 非聊天主形态 | 主入口是代码锚点到右侧解释层，不是输入问题后等待回答。 |
| 稳定锚点 | 解释不能只绑定行号，必须绑定文件快照、结构节点、行范围、代码 hash 和锚点文本。 |
| 首次无门槛 | 内置示例和预置解释层必须允许用户不配置 API Key 就体验核心界面。 |
| 全量目标保留 | 跨文件认知地图、插件入口、协作、云同步等保留为 P1/P2/Future，不进入 MVP 闭环。 |

## 8. 不建议的 MVP 定位

不建议把 CodeReader MVP 定义成：

- 一个 VS Code 插件；
- 一个网页代码解释器；
- 一个 ChatGPT 包装壳；
- 一个只解释选中代码片段的小工具；
- 一个纯代码审查工具。

这些形态都可以成为后续能力或渠道，但不应该主导 CodeReader 的产品定义。

## 9. 当前实施状态

截至 2026 年 6 月 11 日，MVP 已完成单文件解释闭环、SQLite 持久化、结构化 LLM 生成、变更检测、真实项目树和第一批多语言扩展。当前实现阶段对应开发顺序中的 Task 11：

- 通过文件名、路径、语言和能力信号确定性识别入口、配置、核心业务、数据层、样式、测试和说明文件；
- 生成并持久化轻量推荐阅读路径，每一步说明推荐原因；
- 复用解释节点的阅读状态，按推荐文件聚合未读、已读、已理解和有疑问进度；
- 提供 `app.ts -> login-controller.ts -> user-store.ts` 三文件内置示例，无需 API Key 即可体验；
- 继续遵守 MVP 边界，不实现完整跨文件知识图谱，不把 LLM 作为项目地图显示的前置条件。

Task 11 完成后进入 Task 12：安装包、重启恢复、真实 API 调用和 MVP 总验收。

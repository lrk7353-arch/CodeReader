# CodeReader 需求文档

版本：v0.4  
产品形态：独立桌面端 AI 代码阅读 IDE  
MVP 主线：本地文件/项目打开 + IDE 式代码阅读 + 持久化解释层 + 用户理解状态 + 变更过期提示

---

## 1. 需求背景

用户通过 AI 生成代码后，经常面临以下困境：

- 代码可以运行，但用户不知道具体实现逻辑；
- 用户不知道哪些代码是关键逻辑，哪些只是辅助代码；
- 用户不知道某行代码依赖什么、影响什么；
- 用户不知道代码中是否存在安全、异常、性能或边界风险；
- 用户每次向 AI 询问代码含义，得到的解释都不稳定；
- 用户无法把理解过程沉淀为项目知识。

CodeReader 的需求核心是：

> 让用户在类似 IDE 的稳定界面中，逐行、逐块、逐函数、逐文件阅读和掌握 AI 生成代码。

---

## 2. 用户故事

### 2.1 打开项目

作为一个使用 AI 生成项目的用户，  
我希望能用 CodeReader 打开本地项目文件夹，  
以便像使用 IDE 一样浏览代码文件。

验收标准：

- 用户可以选择本地文件夹；
- 系统可以展示项目文件树；
- 用户可以点击文件并在代码区打开；
- 系统可以记录最近打开的项目。

### 2.2 查看文件概览

作为用户，  
我希望打开一个代码文件后先看到文件级说明，  
以便理解这个文件在项目中的职责。

验收标准：

- 系统展示文件职责；
- 系统展示主要函数/类；
- 系统展示主要依赖；
- 系统展示文件风险摘要；
- 解释结果可以保存并再次加载。

### 2.3 逐行阅读代码

作为用户，  
我希望点击某一行代码时，右侧看到这行代码的解释，  
以便理解它的代码层意义、局部组成意义、项目全局意义和风险。

验收标准：

- 点击代码行后，右侧解释面板切换到对应行；
- 解释内容包含代码层意义、局部组成意义、项目全局意义、潜在风险；
- 当前行在代码区高亮；
- 与该行相关的依赖行或影响行可以被标记或查看；
- 解释内容从本地数据库读取，不必每次重新生成。

### 2.4 理解代码块

作为用户，  
我希望可以选择一段代码块，  
以便理解这一组代码共同完成什么任务。

验收标准：

- 支持选择多行；
- 系统识别代码块范围；
- 解释面板展示代码块目标、执行流程、输入输出和风险；
- 代码块解释与行级解释可以互相跳转。

### 2.5 理解函数

作为用户，  
我希望点击函数名或函数区域时看到函数级解释，  
以便理解函数的输入、输出、副作用和调用关系。

验收标准：

- 系统识别函数边界；
- 展示函数职责；
- 展示参数含义；
- 展示返回值；
- 展示调用的其他函数；
- 展示被哪些代码调用；
- 展示异常和边界场景。

### 2.6 保存理解状态

作为用户，  
我希望可以把某行或某段标记为“已理解”“有疑问”“有风险”，  
以便形成自己的阅读进度。

验收标准：

- 每个解释节点可以设置状态；
- 状态保存到本地；
- 文件树可以显示整体理解进度；
- 再次打开项目时状态仍然存在。

### 2.7 代码变更后更新解释

作为用户，  
我希望代码变化后系统能告诉我哪些解释已经过期，  
以便只更新受影响部分。

验收标准：

- 系统计算文件 hash 或 diff；
- 系统识别变更行；
- 受影响解释节点标记为过期；
- 未受影响解释继续保留；
- 用户可以手动触发重新解释。

### 2.8 追问某行代码

作为用户，  
我希望在看不懂某行时可以进一步追问，  
但追问不应破坏主界面的结构化阅读体验。

验收标准：

- 追问入口位于解释面板内；
- 追问上下文自动绑定当前行/代码块/函数；
- 追问结果可以选择是否保存为补充解释；
- 追问不是主产品入口。

### 2.9 第一公里阅读引导

作为第一次打开陌生项目的用户，  
我希望 CodeReader 告诉我应该先读哪些文件，  
以免面对大量文件时不知道从哪里开始。

验收标准：

- 打开项目后系统展示推荐阅读路径；
- 推荐路径说明每个文件为什么值得先读；
- 系统展示入口文件、配置文件、核心业务文件的简要分类；
- 用户可以从推荐路径直接打开文件并开始阅读；
- 系统显示阅读进度。

### 2.10 解释质量反馈

作为用户，  
我希望可以告诉系统某条解释“有帮助”或“不对劲”，  
以便把我的判断沉淀到解释层，而不是被动接受 AI 的说法。

验收标准：

- 解释面板提供“有帮助”“不对劲”“重新解释”入口；
- 用户可以选择“更白话”或“更详细”的解释方式；
- 用户反馈保存到本地；
- 被标记“不对劲”的解释可以局部重新生成；
- 用户反馈不覆盖原解释，而是形成解释版本或反馈记录。

## 4. 非功能需求

### 4.1 性能

| 项目 | 要求 |
|---|---|
| 打开中小型项目 | 5 秒内完成文件树加载 |
| 打开单个文件 | 1 秒内展示代码内容 |
| 加载已保存解释 | 1 秒内显示 |
| 行点击响应 | 200ms 内切换解释 |
| 单文件首次解释 | 可异步执行，并显示进度 |

### 4.2 稳定性

- LLM 生成失败不应导致应用崩溃；
- 文件过大时应提示用户分段分析；
- 数据库损坏时应提供备份/重建机制；
- 解释层与代码不匹配时必须明确提示。
- 解释生成失败或反馈“不对劲”时，应允许局部重新解释。
- 面向用户的可信提示应使用可读标签和解释理由，而不是只展示数字置信度。

### 4.3 隐私

- 默认本地保存项目索引和解释层；
- 默认本地保存用户反馈和阅读状态；
- 用户明确触发分析时才向模型服务发送代码；
- 用户应知道哪些文件会被发送；
- 支持忽略敏感文件；
- API Key 存储必须加密或调用系统安全存储。

### 4.4 可维护性

- 解释生成 prompt 需要版本化；
- 解释数据结构需要版本化；
- 代码解析模块与 LLM 模块解耦；
- UI 组件与数据模型解耦。

## 5. MVP 范围

### 5.1 MVP 必须实现

- 独立桌面端应用；
- 打开本地项目或单文件；
- 内置示例项目体验区；
- 推荐阅读路径和轻量项目文件地图；
- 文件树；
- 代码阅读区；
- 右侧解释面板；
- JavaScript/TypeScript 优先支持；
- 文件级、函数级、代码块级、关键行按需解释；
- 前置知识/学习提示；
- 白话/详细解释模式；
- 解释层本地持久化；
- 点击行展示稳定解释；
- 文件变化后标记解释过期；
- 变更摘要和局部重新解释；
- 用户理解状态标记；
- 解释反馈入口；
- 无 API Key 时可通过内置示例体验核心界面。

### 5.2 MVP 明确不做

- 不做 VS Code 插件作为主产品；
- 不做完整代码编辑器；
- 不做多人协作；
- 不做 PR 审查平台；
- 不做云端项目托管；
- 不做所有语言支持；
- 不做自动修改代码为主的 coding agent。

## 7. 角色与权限

MVP 仅考虑单用户本地使用。

未来可扩展角色：

| 角色 | 权限 |
|---|---|
| Owner | 管理项目、解释层、模型配置 |
| Reader | 阅读解释、标记理解状态 |
| Reviewer | 添加风险评注、审阅建议 |
| Team Admin | 管理团队共享解释层 |

## 8. 验收总标准

CodeReader MVP 可以被认为完成，当它满足以下条件：

1. 用户可以下载安装并启动独立桌面应用；
2. 用户可以打开本地项目；
3. 用户可以在类似 IDE 的界面中浏览代码；
4. 用户可以对一个文件生成解释层；
5. 用户点击某一行时，右侧展示与该行对应的稳定解释；
6. 用户关闭并再次打开项目后，解释仍然存在；
7. 用户修改代码后，系统能标记解释过期；
8. 系统能展示变更摘要并支持局部重新解释；
9. 用户可以标记某行“已理解/有疑问/有风险”；
10. 用户可以反馈某条解释“有帮助/不对劲”；
11. 用户打开陌生项目时能看到推荐阅读路径；
12. 用户首次启动时不配置 API Key，也能通过内置示例体验行/块/函数解释映射；
13. 用户能通过 CodeReader 完成一个关键文件的系统阅读。

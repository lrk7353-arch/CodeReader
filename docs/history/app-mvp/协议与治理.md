# CodeReader 协议与治理

版本：v0.4  
适用范围：解释层数据协议、持久化治理、隐私安全、版本管理、开源/商业协议策略

---

## 1. 治理目标

CodeReader 的核心资产不是源代码本身，而是围绕源代码生成的解释层、关系层、风险层和用户理解状态。

因此，需要从一开始定义清楚：

- 解释数据如何保存；
- 解释数据如何与代码稳定映射；
- 解释数据如何版本化；
- 解释数据如何导出和迁移；
- 用户反馈如何保存和用于解释改进；
- 代码变更记录如何与解释状态对应；
- 用户代码如何保护；
- AI 生成内容如何声明；
- 后续开源和商业化如何处理。

---

## 2. 解释层数据协议

### 2.1 协议名称

建议命名为：

```text
CodeReader Explanation Layer Protocol
```

简称：

```text
CREL
```

### 2.2 协议目标

CREL 用于描述代码与解释之间的稳定映射关系。

本协议面向全量产品能力设计，但 MVP 只实现最小可用子集：解释节点、稳定锚点、用户阅读状态、解释反馈、代码变更记录和轻量阅读路径。跨文件知识图谱、团队共享和插件协议保留为后续扩展，不作为 MVP 验收前置条件。

它应支持：

- 文件级解释；
- 函数级解释；
- 代码块级解释；
- 行级解释；
- 风险说明；
- 用户阅读状态；
- 用户解释反馈；
- 推荐阅读路径；
- 项目文件地图；
- 解释版本；
- 代码变更后的有效性状态。

## 3. 解释节点协议

### 3.1 节点结构

```json
{
  "id": "exp_node_001",
  "project_id": "project_001",
  "file_id": "file_001",
  "file_path": "src/auth/login.ts",
  "file_hash": "sha256:file-content",
  "snapshot_id": "snapshot_001",
  "node_type": "line",
  "start_line": 37,
  "end_line": 37,
  "code_hash": "sha256:...",
  "ast_node_id": "optional_ast_node_id",
  "symbol_id": "optional_symbol_id",
  "anchor_text": "const user = await getUser(id)",
  "surface_meaning": "根据 id 异步获取用户对象。",
  "actual_meaning": "运行时会等待 getUser 的异步结果。",
  "relational_meaning": "依赖前面获得的 id，后续权限判断会使用 user。",
  "global_role": "为登录流程提供用户身份数据。",
  "code_level_meaning": "根据 id 异步获取用户对象。",
  "local_composition_meaning": "为当前登录函数提供后续权限判断所需的 user。",
  "project_role_meaning": "服务于认证流程中的身份确认步骤。",
  "prior_knowledge": "需要理解 async/await 和异步查询结果。",
  "risks": [],
  "learning_note": "需要理解 async/await。",
  "review_suggestion": "确认是否处理 user 为空的情况。",
  "relations": [],
  "trust_label": "context_needed",
  "trust_reason": "这行本身含义明确，但安全性取决于后续是否处理 user 为空。",
  "raw_confidence": 0.82,
  "display_mode": "plain",
  "status": "valid",
  "schema_version": "crel-0.1",
  "prompt_version": "line-exp-v0.1",
  "model_info": {
    "provider": "unknown",
    "model": "unknown"
  },
  "created_at": "2026-06-08T00:00:00Z",
  "updated_at": "2026-06-08T00:00:00Z"
}
```

### 3.2 node_type

| 类型 | 说明 |
|---|---|
| project | 项目级解释 |
| module | 模块级解释 |
| file | 文件级解释 |
| symbol | 函数/类/变量等符号解释 |
| block | 代码块解释 |
| line | 行级解释 |
| risk | 风险节点 |
| learning | 学习提示节点 |
| reading_path | 推荐阅读路径节点 |
| project_map | 项目地图节点 |

### 3.3 status

| 状态 | 说明 |
|---|---|
| valid | 当前解释有效 |
| stale | 代码变化，解释可能过期 |
| invalid | 解释与代码不再匹配 |
| new_unexplained | 新增代码尚未解释 |
| deleted | 原代码已删除，解释仅保留为历史 |
| user_modified | 用户修改过解释 |
| ai_regenerated | AI 重新生成过 |
| archived | 历史解释，默认不展示 |

`raw_confidence` 只能作为内部诊断信息，不应在界面中裸露为主要判断依据。面向用户应展示 `trust_label` 与 `trust_reason`，例如“解释相对明确”“这里依赖上下文较多”“建议重点检查”。

### 3.4 稳定锚点要求

解释节点不能只依赖行号。MVP 中每条解释至少应能追溯到：

- `project_id`；
- `file_id` / `file_path`；
- `file_hash`；
- `snapshot_id`；
- `node_type` / `ast_node_id` / `symbol_id`；
- `start_line` / `end_line`；
- `code_hash`；
- `anchor_text`；
- `status`；
- `created_at` / `updated_at`。

其中 `anchor_text` 保存目标代码片段的短文本，用于在行号漂移但代码内容仍可匹配时辅助迁移解释；`file_hash` 与 `snapshot_id` 用于判断解释对应的是哪个文件版本。

## 4. 关系协议

### 4.1 relation_type

| 类型 | 说明 |
|---|---|
| depends_on | 当前节点依赖目标节点 |
| used_by | 当前节点被目标节点使用 |
| calls | 当前节点调用目标函数 |
| called_by | 当前节点被目标调用 |
| controls | 当前节点控制目标节点是否执行 |
| validates | 当前节点校验目标节点所需数据 |
| transforms | 当前节点转换数据供目标使用 |
| handles_error_for | 当前节点处理目标节点可能产生的错误 |
| introduces_risk | 当前节点引入风险 |
| mitigates_risk | 当前节点缓解风险 |

### 4.2 关系节点示例

```json
{
  "id": "rel_001",
  "source_node_id": "exp_line_37",
  "target_node_id": "exp_line_43",
  "relation_type": "used_by",
  "description": "第 43 行使用第 37 行获取到的 user 对象进行权限判断。",
  "confidence": 0.76
}
```

## 5. 用户阅读状态协议

用户对解释节点的理解状态应与 AI 解释分开保存。

### 5.1 状态类型

| 状态 | 说明 |
|---|---|
| unread | 未阅读 |
| read | 已阅读 |
| understood | 已理解 |
| confused | 有疑问 |
| risky | 用户认为有风险 |
| accepted_risk | 用户接受风险 |
| needs_test | 需要补充测试 |
| needs_refactor | 需要重构 |

### 5.2 状态示例

```json
{
  "explanation_id": "exp_line_37",
  "state": "understood",
  "user_note": "这里我理解为先取用户，再检查权限。",
  "updated_at": "2026-06-08T00:00:00Z"
}
```

## 6. 解释反馈协议

用户反馈应与 AI 解释和阅读状态分开保存。它用于表达“这条解释对用户是否有帮助”，而不是表达用户是否已经读懂代码。

### 6.1 feedback_type

| 类型 | 说明 |
|---|---|
| helpful | 用户认为解释有帮助 |
| suspicious | 用户认为解释不对劲 |
| too_vague | 解释太泛 |
| too_technical | 解释太技术化 |
| needs_plain_language | 需要更白话 |
| needs_more_detail | 需要更多技术细节 |
| regenerate_requested | 请求重新解释 |
| user_note | 用户补充说明 |

### 6.2 反馈示例

```json
{
  "explanation_id": "exp_line_37",
  "feedback_type": "suspicious",
  "user_note": "这里似乎没有处理 user 为空，但解释说风险很低。",
  "created_at": "2026-06-08T00:00:00Z"
}
```

治理要求：

- 反馈不得覆盖原解释；
- 局部重新解释应生成新版本；
- 反馈可以用于后续 prompt 优化；
- 用户应能删除自己的反馈和笔记。

## 7. 代码变更协议

持久化解释层必须能够说明解释与代码当前版本的关系。

### 7.1 change_record

```json
{
  "file_path": "src/auth/login.ts",
  "before_hash": "sha256:...",
  "after_hash": "sha256:...",
  "added_lines": 12,
  "modified_lines": 5,
  "deleted_lines": 2,
  "affected_explanations": ["exp_line_37", "exp_block_2"],
  "summary": "新增 1 个校验分支，修改登录响应构造，2 条解释可能过期。",
  "created_at": "2026-06-08T00:00:00Z"
}
```

### 7.2 状态治理

代码变更后：

- 未变化解释保持 `valid`；
- 附近代码变化但目标片段未变，标记 `stale`；
- 目标代码片段变化，标记 `invalid`；
- 新增代码标记 `new_unexplained`；
- 删除代码对应解释标记 `deleted` 或 `archived`。

## 8. 版本治理

### 8.1 必须版本化的对象

| 对象 | 是否必须版本化 | 原因 |
|---|---|---|
| 解释层 schema | 是 | 防止数据结构变化导致旧解释不可读 |
| Prompt 模板 | 是 | 不同 prompt 会生成不同风格解释 |
| 模型信息 | 是 | 不同模型输出质量不同 |
| 文件 hash | 是 | 判断解释是否仍然有效 |
| AST 节点 hash | 是 | 支持代码变化后的解释迁移 |
| 用户修改 | 是 | 区分 AI 解释和用户认知沉淀 |
| 用户反馈 | 是 | 记录解释质量判断和重解释原因 |
| 变更记录 | 是 | 解释为什么变为 stale/invalid |
| 项目地图/阅读路径 | 是 | 项目结构变化会影响推荐路径 |

### 8.2 版本命名建议

```text
schema_version: crel-0.1
prompt_version: line-exp-v0.1
app_version: codereader-0.1.0
```

## 9. 隐私与安全治理

### 9.1 默认原则

CodeReader 应坚持本地优先：

- 不默认上传完整项目；
- 不默认上传隐藏文件；
- 不默认上传 `.env`、密钥、证书、构建产物；
- 用户触发分析时明确说明发送范围；
- 解释层默认保存在本地。
- 用户反馈和阅读状态默认保存在本地。

### 9.2 敏感文件忽略规则

默认忽略：

```text
.env
.env.*
*.pem
*.key
*.crt
node_modules/
dist/
build/
.git/
coverage/
.DS_Store
```

### 9.3 API Key 管理

要求：

- 不明文存储 API Key；
- 优先调用系统安全存储；
- 日志中不得输出 API Key；
- 用户可以随时删除模型配置。

### 9.4 AI 解释免责声明

产品应明确提示：

- AI 解释可能存在错误；
- 风险提示不等于完整安全审计；
- 用户应对最终代码负责；
- 对关键业务、安全、支付、隐私相关代码，需要人工或专业审查。

## 10. 数据导出协议

用户应可以导出解释层，以防止数据被锁定在 CodeReader 中。

### 10.1 导出格式

MVP 后建议支持：

- JSON：完整解释层；
- JSON：用户反馈、阅读状态、变更记录、项目地图；
- Markdown：面向阅读的人类文档；
- HTML：可分享项目说明书；
- SQLite backup：完整本地数据备份。

### 10.2 Markdown 导出结构

```markdown
# src/auth/login.ts 解释文档

## 文件职责

...

## 函数说明

### loginUser

...

## 逐行解释

### 第 37 行

代码：

```ts
const user = await getUser(id)
```

代码层意义：...
局部组成意义：...
项目全局意义：...
前置知识：...
可信提示：...
风险：...
```

## 11. 开源与商业协议建议

### 11.1 早期建议

如果希望吸引开发者参与，可以考虑：

- 核心解释层协议开源；
- 本地单机版部分开源；
- 云同步、团队协作、企业能力商业化。

### 11.2 可选协议策略

| 模式 | 说明 | 适合情况 |
|---|---|---|
| MIT | 最大程度开放 | 追求生态传播 |
| Apache-2.0 | 开放且带专利授权 | 追求企业采用 |
| AGPL | 要求服务端改动开源 | 防止云厂商直接套壳 |
| Open Core | 核心开源，高级能力商业化 | 适合长期产品化 |
| Proprietary | 完全闭源 | 适合先快速商业验证 |

### 11.3 推荐路线

建议采用 **Open Core** 思路：

| 部分 | 策略 |
|---|---|
| CREL 解释层协议 | 开源 |
| 基础桌面端 | 可考虑开源或源代码可见 |
| 高级项目图谱 | 商业版 |
| 团队协作 | 商业版 |
| 云同步 | 商业版 |
| 企业私有部署 | 商业版 |

## 12. 插件治理

由于 CodeReader 主产品不是 VS Code 插件，因此插件应被治理为生态入口，而不是核心产品。

### 12.1 插件职责

插件只做：

- 从 VS Code 打开当前项目到 CodeReader；
- 从 VS Code 打开当前文件到 CodeReader；
- 显示当前文件是否已有 CodeReader 解释层；
- 读取本地解释层摘要。

### 12.2 插件不做

插件不应承担：

- 完整解释层管理；
- 完整项目认知地图；
- 主阅读流程；
- 复杂状态治理；
- 全部用户体验。

---

## 13. 数据所有权

CodeReader 应明确：

- 用户拥有自己的源代码；
- 用户拥有本地生成和保存的解释层；
- 用户拥有自己的反馈、笔记、阅读状态和导出数据；
- 用户可以导出、删除解释层；
- 如果使用云服务，需明确同步范围和存储位置；
- 不应将用户代码用于模型训练，除非用户明确授权。

| API Key 安全存储 | 必须 |
| 解释反馈协议 | 必须 |
| 变更记录协议 | 必须 |
| 推荐阅读路径协议 | 必须 |
| 解释导出 | P1 |
| 云同步协议 | P2 |
| 团队权限 | P2 |
| 插件协议 | P2 |
| 组织级审计 | P3 |

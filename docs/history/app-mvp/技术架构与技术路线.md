# CodeReader 技术架构与技术路线

版本：v0.4  
架构定位：独立桌面端本地优先应用  
MVP 技术主线：Tauri + React + Monaco Editor + SQLite + Tree-sitter + LLM API

---

## 1. 技术目标

CodeReader 的技术架构必须服务于以下产品目标：

1. 提供类似 IDE 的代码阅读体验；
2. 建立代码行、代码块、函数、文件与解释数据的稳定映射；
3. 将解释层持久化保存到本地；
4. 支持代码变更后的增量过期识别；
5. 支持用户对解释进行反馈、质疑和局部重新解释；
6. 支持第一公里阅读引导、推荐阅读路径和轻量项目认知地图；
7. 支持后续扩展为项目级认知地图、多语言解析、云端同步和团队协作。

因此，CodeReader 不应以 VS Code 插件作为核心架构起点，而应以独立桌面应用作为主线。

本轮启动前检查只收束 MVP 实现，不删除全量产品目标。跨文件认知地图、多语言解析、云端同步、团队协作和插件入口可以保留在 P1/P2/Future 路线中，但不得进入 MVP 的完成定义。

---

## 2. 推荐技术栈

| 层级 | 技术 | 作用 | MVP 是否采用 |
|---|---|---|---|

## 4. 核心模块设计

### 4.1 Project Service

负责项目级管理。

能力：

- 打开本地项目；
- 建立项目 ID；
- 扫描文件树；
- 应用忽略规则；
- 保存项目元信息；
- 维护最近打开项目。

项目 ID 建议由以下信息生成：

```text
project_path + git_remote(optional) + created_at
```

### 4.2 File Service

负责文件读取、变更检测和状态维护。

能力：

- 读取文件内容；
- 计算文件 hash；
- 判断文件是否变化；
- 保存文件版本；
- 建立文件路径与解释层映射。

### 4.3 Code Parse Service

负责从源代码中提取结构。

MVP 优先支持：

- JavaScript；
- TypeScript；
- Python 可作为第二阶段支持。

解析目标：

- 文件语言；
- import/export；
- 函数声明；
- 类声明；
- 代码块；
- 行号范围；
- 变量定义和引用；
- AST 节点 hash。

### 4.4 Context Builder Service

负责为不同解释目标选择必要上下文。它不是简单截取整文件，而是根据解释任务动态组装上下文，控制成本并降低误解。

核心原则：

- 当前解释目标必须完整出现；
- 所在函数或代码块优先于整文件；
- 静态分析得到的定义、引用、调用关系优先于 LLM 猜测；
- 大文件先文件级/函数级，再按需行级展开；
- 上下文来源应可追踪，便于解释面板说明“这条解释参考了哪些代码”。

上下文策略：

| 解释目标 | 上下文组成 |
|---|---|
| 单行解释 | 当前行 + 所在函数 + 相关变量定义 + 相关调用处摘要 + 前后少量行 |
| 多行/代码块解释 | 代码块 + 所在函数 + 输入输出变量 + 控制流关系 |
| 函数解释 | 函数体 + 参数/返回值 + 被调用函数摘要 + 调用该函数的位置 |
| 文件解释 | 文件结构 + import/export + 核心函数摘要 + 文件角色推断 |
| 项目/模块解释 | 文件树 + 入口文件 + 模块摘要 + 依赖关系 |

文件规模策略：

| 文件规模 | 默认解释策略 |
|---|---|
| 0-200 行 | 文件级 + 函数级 + 关键行级解释 |
| 200-800 行 | 文件级 + 函数级，用户点击后展开行级 |
| 800-2000 行 | 先生成模块/函数索引，只解释关键入口、调用链和风险区域 |
| 2000 行以上 | 不默认逐行解释，提示拆分阅读并生成阅读路径 |

### 4.5 Explanation Service

负责生成、保存、读取解释层。

核心职责：

- 构造 prompt；
- 调用 LLM；
- 要求模型输出结构化 JSON；
- 校验返回格式；
- 保存解释节点；
- 根据用户当前选择读取解释。
- 支持白话模式和详细模式；
- 支持局部重新解释；
- 支持把数字置信度转换为用户可读的可信提示。

解释节点类型：

```text
project_explanation
file_explanation
symbol_explanation
block_explanation
line_explanation
risk_explanation
learning_note
```

### 4.6 Explanation Feedback Service

负责保存用户对解释的判断。CodeReader 不应依赖“AI 解释后再由 AI 自检”的闭环，而应让用户参与解释层演进。

反馈类型：

- helpful：这条解释有帮助；
- suspicious：这条解释不对劲；
- too_vague：解释太泛；
- too_technical：解释太技术化；
- needs_plain_language：需要更白话；
- needs_more_detail：需要更多细节；
- regenerate_requested：用户请求重新解释；
- user_note：用户补充自己的理解。

这些反馈应进入本地数据库，用于：

- 调整解释显示；
- 触发局部重新解释；
- 形成解释版本历史；
- 后续优化 prompt。

### 4.7 Diff & Expiration Service

负责代码变化后的解释有效性判断。

策略：

1. 文件 hash 不变：解释全部有效；
2. 文件 hash 变化但 AST 节点 hash 不变：对应解释可继续有效；
3. AST 节点变化：节点下解释标记为过期；
4. 行号变化但代码片段 hash 不变：尝试迁移解释；
5. 代码片段变化：标记为需重新解释。

同时生成变更摘要：

```text
自上次阅读以来：
- 新增 3 个函数；
- 修改 12 行；
- 删除 1 个代码块；
- 5 条解释可能过期；
- 2 段新增代码尚未解释。
```

MVP 使用文件 hash、行级 diff 和 AST 节点 hash；P1 可接入 Git diff，P2 再进入 commit/PR 级解释。

### 4.8 Project Map / Reading Path Service

负责第一公里阅读引导和轻量项目认知地图。

MVP 不做复杂可视化图谱，先做文本化能力：

- 项目文件地图：识别入口、配置、业务逻辑、样式、数据层；
- 推荐阅读路径：告诉用户先读哪些文件，以及为什么；
- 模块关系摘要：用自然语言描述几个核心模块如何协作；
- 阅读进度：已读、已理解、有疑问、有风险的比例。

### 4.9 User State Service

负责保存用户阅读过程。

状态包括：

- 未读；
- 已读；
- 已理解；
- 有疑问；
- 有风险；
- 已确认风险可接受；
- 需要测试；
- 需要重构。

```sql
  id TEXT PRIMARY KEY,
```

```sql
  id TEXT PRIMARY KEY,
```

```sql
  end_line INTEGER NOT NULL,

  parent_node_id TEXT

```

### 5.5 explanation_nodes

```sql
CREATE TABLE explanation_nodes (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  file_id TEXT NOT NULL,
  snapshot_id TEXT NOT NULL,
  code_node_id TEXT,
  explanation_type TEXT NOT NULL,
  start_line INTEGER,
  end_line INTEGER,
  code_level_meaning TEXT,
  local_composition_meaning TEXT,
  project_role_meaning TEXT,
  surface_meaning TEXT,
  actual_meaning TEXT,
  relational_meaning TEXT,
  global_role TEXT,
  prior_knowledge TEXT,
  risk_summary TEXT,
  learning_note TEXT,
  review_suggestion TEXT,
  trust_label TEXT,
  trust_reason TEXT,

  display_mode TEXT,

  schema_version TEXT NOT NULL,
  prompt_version TEXT NOT NULL,
```

`raw_confidence` 仅作为内部诊断字段，不应作为主要用户界面文案。面向用户应展示 `trust_label` 与 `trust_reason`，例如“解释相对明确”“这里依赖上下文较多，建议结合调用处一起看”。

### 5.6 explanation_targets

```sql
  ast_hash TEXT,
```

解释目标类型：

```text

range
block
function
class
file
```

## 6. 解释生成流程

### 6.1 首次分析流程

```text
打开项目
  ↓
扫描文件树
  ↓
用户选择文件
  ↓
读取文件内容
  ↓
Tree-sitter 解析 AST
  ↓
提取函数/代码块/行级节点
  ↓
Context Builder 构造文件级上下文
  ↓
调用 LLM 生成文件级解释
  ↓
Context Builder 分批构造函数/代码块/关键行上下文
  ↓
调用 LLM 生成函数/代码块/关键行解释
  ↓
校验结构化输出
  ↓
保存到 SQLite
  ↓
右侧解释面板展示
```

### 6.2 再次打开流程

```text
打开项目
  ↓
读取文件 hash
  ↓
与 last_analyzed_hash 对比
  ↓
无变化：直接加载解释层
有变化：标记解释过期并提示更新
```

### 6.3 增量更新流程

```text
检测文件变更
  ↓
计算 diff
  ↓
重新解析 AST
  ↓
匹配旧节点与新节点
  ↓
判断解释有效性
  ↓
迁移可复用解释
  ↓
标记过期解释
  ↓
生成变更摘要
  ↓
仅重新生成变更区域解释
```

## 7. Prompt 与结构化输出

LLM 不应返回自由散文，而应返回结构化数据。

行级解释输出示例：

```json
{
  "line": 37,
  "code": "const user = await getUser(id)",
  "code_level_meaning": "根据 id 异步获取用户对象。",
  "local_composition_meaning": "当前函数会等待 getUser 返回结果，然后把结果交给后续权限判断。",
  "project_role_meaning": "这是认证链路中取得用户身份数据的关键步骤。",
  "prior_knowledge": "需要先知道 async/await 和异步函数返回值。",
  "risks": [
    {
      "type": "null_handling",
      "level": "medium",
      "description": "如果 getUser 返回 null，后续访问 user.role 可能报错。"
    }
  ],
  "review_suggestion": "建议检查后续是否处理 user 为空的情况。",
  "depends_on_lines": [12, 28, 34],
  "affects_lines": [39, 43, 51],
  "trust_label": "context_needed",
  "trust_reason": "这行本身含义明确，但是否安全取决于后续是否处理 user 为空。",
  "display_mode": "plain"
}
```

解释输出应优先使用三层字段：`code_level_meaning`、`local_composition_meaning`、`project_role_meaning`。旧字段 `surface_meaning`、`actual_meaning`、`relational_meaning` 可作为兼容字段保留，但 UI 默认按三层结构展示。

## 8. 本地优先架构

CodeReader 应默认本地优先：

- 项目代码在本地；
- 解释层在本地；
- 阅读状态在本地；
- 用户明确触发时才把必要代码片段发送给 LLM；
- 后续云同步是可选能力，不是 MVP 前提。

本地目录建议：

```text
~/Library/Application Support/CodeReader/        # macOS
%APPDATA%/CodeReader/                            # Windows
~/.config/CodeReader/                            # Linux
```

项目内可选创建：

```text
.codereader/
  project.json
  explanation-export.json
```

MVP 建议默认不污染用户项目目录，除非用户主动导出解释层。

## 9. VS Code 插件的位置

VS Code 插件不作为 MVP 主线。

它在技术路线中的定位是：

| 阶段 | 角色 |
|---|---|
| MVP | 不做，避免产品定位被插件化 |
| P1/P2 | 可作为“从 VS Code 打开 CodeReader”的桥接入口 |
| 成熟期 | 可作为开发者分发渠道，但解释层仍由 CodeReader 管理 |

插件能力可以包括：

- 当前文件发送到 CodeReader；
- 当前项目在 CodeReader 中打开；
- 在 VS Code 中显示 CodeReader 的解释状态；
- 读取 CodeReader 本地解释层。

但插件不应承载完整产品体验。

## 10. 技术演进路线

### 阶段 0：体验原型验证

目标：在不引入完整工程复杂度的情况下，验证“左侧代码 + 右侧解释层”是否真的比聊天式解释更有效。

实现：

- 纯网页 Demo；
- Monaco 展示固定示例代码；
- 右侧展示预置解释 JSON；
- 点击单行、多行、函数切换解释；
- 支持“已理解 / 有疑问 / 不对劲”反馈；
- 准备 3 个示例代码文件；
- 不配置 API Key 也能完整体验核心界面。

不做：

- 不接真实 LLM；
- 不做文件系统；
- 不做 Tauri；
- 不做 SQLite；
- 不做 Git。

验收：目标用户看到 Demo 后，能明确感受到“这个比把代码丢给聊天工具问要清楚”。

### 阶段 1：MVP 独立桌面应用

目标：形成可安装、可打开项目、可稳定阅读的桌面端产品。

实现：

- Tauri 桌面端；
- 文件树；
- JS/TS 解析；
- Tree-sitter 结构锚定；
- SQLite 持久化；
- 文件/函数/代码块/行级解释；
- Context Builder；
- 解释质量反馈；
- 变更摘要；
- 推荐阅读路径；
- 过期标记；
- 用户理解状态。

### 阶段 2：项目级认知能力

目标：从单文件理解扩展到项目级理解。

实现：

- 跨文件依赖图；
- 项目入口识别；
- 推荐阅读路径；
- 关键风险聚合；
- 搜索解释层。

### 阶段 3：多语言与高级分析

## 11. 主要技术风险

| 风险 | 说明 | 应对 |
|---|---|---|
| 行号变化导致解释错位 | 代码修改后行号变化 | 使用 AST 节点 hash + 代码片段 hash 辅助映射 |
| LLM 输出不稳定 | 每次解释不同 | 固定 schema、prompt version、保存结果 |
| 大文件解释成本高 | token 成本和延迟较高 | 分块分析、优先函数级，再下钻行级 |
| 关系识别不准确 | LLM 可能误判依赖关系 | 结合 Tree-sitter/LSP/静态分析 |
| 用户误信解释 | AI 解释可能错误 | 使用可读可信提示、风险提示、模型来源和用户反馈入口 |
| 上下文构造不当 | 上下文太少会误解，太多会浪费且增加幻觉 | 使用 Context Builder 按目标选择上下文 |
| 第一公里体验弱 | 用户打开项目后不知道从哪里开始读 | 提供示例项目、推荐阅读路径和项目文件地图 |
| 隐私风险 | 代码发送给外部模型 | 本地优先、敏感文件过滤、明确授权 |

## 12. 推荐开发顺序

1. 做纯网页体验原型，验证 Monaco + 解释面板 + 预置解释 JSON；
2. 创建 Tauri + React 基础应用；
3. 集成 Monaco Editor；
4. 实现打开本地文件/项目；
5. 实现文件树和代码展示；
6. 接入 Tree-sitter，识别函数、代码块、import/export，并建立行到结构节点的映射；
7. 接入 SQLite；
8. 定义解释层 schema、target、feedback、reading_state、change_record；
9. 保存并加载 mock 解释层；
10. 实现点击单行/多行/函数展示解释；
11. 实现 Context Builder；
12. 接入 LLM 生成文件级、函数级、代码块级和关键行解释；
13. 增加解释反馈、白话/详细模式和局部重新解释；
14. 实现代码变更检测、过期标记和变更摘要；
15. 增加推荐阅读路径、项目文件地图和阅读进度；
16. 打磨 MVP 体验并准备内置示例。

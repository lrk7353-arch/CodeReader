# CodeReader 产品设计文档

版本：v0.4  
产品定位：独立桌面端 AI 代码阅读 IDE  
核心关键词：代码认知、行级解释、稳定映射、持久化解释层、AI 代码所有权

---

## 1. 背景

AI 编程工具正在让越来越多非科班用户具备“通过自然语言生成软件”的能力。用户可以向大模型描述需求，由 AI 生成前端、后端、脚本、插件、自动化流程甚至完整项目。

但新的问题随之出现：

> 用户拥有了生成代码的能力，却不一定拥有理解代码、审阅代码、维护代码和承担代码责任的能力。

对于缺乏系统编程训练的用户来说，AI 生成代码往往具有以下特征：

1. **可运行但不可理解**：项目看似能跑，但用户不知道每一行为什么存在。
2. **可交付但不可掌控**：功能已经实现，但用户难以判断架构是否合理、风险是否可接受。
3. **可迭代但不可维护**：每次改动都需要继续依赖 AI，而用户自身无法形成稳定认知。
4. **可审查但不可内化**：其他 AI 或代码审查工具可以发现问题，但用户未必真正理解问题。

CodeReader 的出现，是为了填补 AI 编程时代的关键断层：

> AI 降低了写代码门槛，但没有同步降低理解代码的门槛。

---

## 2. 产品目的

CodeReader 的目的不是替代 IDE 写代码，也不是替代 AI 生成代码，而是提供一个专门的代码阅读环境，帮助用户把 AI 生成的代码变成可理解、可审阅、可维护、可负责的知识结构。

### 2.1 核心目的

- 帮助用户逐行理解 AI 生成代码；
- 帮助用户理解一行代码与其他行、函数、文件、模块之间的关系；
- 帮助用户识别代码中的潜在风险和不确定性；
- 帮助用户形成对项目的整体认知；
- 帮助用户把一次性 AI 解释沉淀为长期可维护的项目解释层。

### 2.2 不是要做什么

CodeReader 不应被设计成：

- 普通聊天式代码问答工具；
- 单次生成解释的网页工具；
- 只解释代码片段的轻量应用；
- 仅供专业程序员审查 bug 的代码审查工具；
- 依附于 VS Code 的普通 AI 插件。

它的核心形态应是：

> 一个独立的、以代码阅读为核心的 AI 代码认知 IDE。

## 3. 目标用户

### 3.1 核心用户

| 用户类型 | 典型特征 | 主要痛点 |
|---|---|---|
| AI 编程初学者 | 能用 AI 生成代码，但基础薄弱 | 看不懂生成结果，不知道是否可靠 |
| 文科/非科班背景创作者 | 能表达产品需求，但缺乏工程训练 | 项目变成黑箱，难以负责 |
| 独立开发者 | 使用 Cursor、Claude Code、Codex、Cline 等工具快速开发 | 代码增长后认知跟不上 |
| 产品经理/创业者 | 通过 AI 搭建原型或 MVP | 不知道项目真实实现方式和风险 |
| 编程学习者 | 想通过真实项目反向学习代码 | 缺乏逐行、结构化、上下文式讲解 |

### 3.2 用户心智

用户并不一定想成为专业工程师，但希望做到：

- 我知道这个项目大概如何工作；
- 我知道重要文件和函数分别负责什么；
- 我知道每一行关键代码为什么存在；
- 我知道哪些地方可能有风险；
- 我可以在 AI 帮助下逐渐掌握项目，而不是永远依赖 AI。

## 4. 产品核心价值

### 4.1 价值主张

> AI 写代码，CodeReader 帮你真正读懂并掌握代码。

### 4.2 更完整的表述

> CodeReader 为 AI 生成代码提供一层稳定、持久、可导航的解释层，让用户能够在 IDE 式界面中逐行理解代码的代码层意义、局部组成意义、项目全局意义和潜在风险，从而重新获得对项目的认知权和责任感。

## 5. 产品形态

### 5.1 总体形态

CodeReader 应是一个独立桌面端应用，界面类似轻量 IDE：

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶部工具栏：打开项目 / 分析文件 / 解释状态 / 搜索 / 设置         │
├───────────────┬───────────────────────────┬─────────────────┤
│ 文件树         │ 代码阅读区                 │ 解释面板          │
│ Project Tree  │ Code View                  │ Explanation     │
│               │ 行号 / 语法高亮 / 选择      │ 与当前行稳定映射   │
│               │ 当前行高亮 / 关系标记        │ 三层解释/风险/反馈  │
├───────────────┴───────────────────────────┴─────────────────┤
│ 底部状态栏：索引状态 / 解释版本 / 模型来源 / 当前文件风险级别     │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 左侧：项目与文件结构

左侧展示项目目录结构，类似 VS Code/PyCharm 的文件树。

核心能力：

- 打开本地项目目录；
- 展示文件树；
- 标记文件解释状态；
- 标记文件风险等级；
- 标记解释是否过期；
- 支持按语言、模块、状态筛选文件。

文件状态示例：

| 状态 | 含义 |
|---|---|
| 未分析 | 该文件尚未生成解释层 |
| 已解释 | 该文件已有稳定解释层 |
| 部分过期 | 文件局部代码变更，部分解释需更新 |
| 已过期 | 文件变更较大，解释层需重新生成 |
| 高风险 | 文件包含较多安全、异常或架构风险 |

### 5.3 中间：代码阅读区

中间是产品主体，展示原始代码文件。

MVP 必须具备：

- 行号；
- 语法高亮；
- 当前行选中；
- 多行代码块选中；
- 解释状态标记；
- 风险标记。

后续可加强：

- 函数/类/模块折叠；
- 行间关系标记。

代码区不是为了高强度编码，而是为了高质量阅读。MVP 可以允许只读或轻编辑，重点是稳定阅读和解释映射。

### 5.4 右侧：解释面板

右侧解释面板是 CodeReader 的核心。

当用户点击某一行时，右侧应展示该行的结构化解释。v0.3 建议将解释收敛为“三层核心解释 + 可展开辅助信息”，避免每一行都堆叠过多术语。

三层核心解释：

| 层级 | 内容 | 面向用户的问题 |
|---|---|---|
| 代码层意义 | 从代码规则、语法和写法解释这一行做了什么 | 这行代码写了什么？ |
| 局部组成意义 | 解释它在当前函数、代码块或模块中承担什么作用 | 它在这段逻辑里干什么？ |
| 项目全局意义 | 解释它和项目流程、业务目标、架构位置的关系 | 它对整个项目有什么意义？ |

可展开辅助信息：

| 解释维度 | 内容 |
|---|---|
| 潜在风险 | 空值、异常、权限、安全、性能、边界条件等风险 |
| 前置知识 | 用户理解这一行前需要知道的概念 |
| 审阅建议 | 是否需要修改、测试、追问或保留 |
| 相关代码 | 依赖哪些行、影响哪些行、关联哪些函数 |
| 可信提示 | 用白话说明解释是否明确，避免只展示难懂的数字置信度 |

当用户选择代码块、函数或文件时，右侧切换到对应粒度的解释。

解释面板默认采用“白话模式”，优先使用类比和日常语言解释；用户可以切换到“详细模式”，查看更技术化的说明、关系、风险和上下文来源。

## 6. 多粒度解释模型

CodeReader 不是只做逐行解释，而是支持多层级认知：

| 粒度 | 解释对象 | 解释目标 |
|---|---|---|
| 行级 | 单行代码 | 解释具体语句的意义、关系、风险 |
| 代码块级 | if/for/try/callback 等代码块 | 解释一组语句共同完成的任务 |
| 函数级 | function/method | 解释输入、输出、副作用、调用关系、失败场景 |
| 文件级 | 单个文件 | 解释文件职责、主要结构、对外接口、依赖 |
| 模块级 | 文件夹/功能模块 | 解释业务能力、内部协作、边界 |
| 项目级 | 整个项目 | 解释架构、数据流、核心风险、阅读路径 |

用户可以从项目级逐步下钻到单行，也可以从一行反向追溯到模块和项目目标。MVP 阶段只要求完成单文件内的文件级、函数级、代码块级和关键行解释闭环；项目级和模块级解释可以先以轻量文本摘要和推荐阅读路径存在，不要求完整知识图谱。

### 6.1 解释深度策略

CodeReader 不应为了“逐行解释”而强行展开所有行。不同代码类型应采用不同解释密度：

| 代码类型 | 默认解释策略 |
|---|---|
| import、常量、简单赋值 | 简短解释，可批量折叠 |
| 函数定义、类定义、API 路由、数据库操作 | 展示三层解释 |
| 条件判断、循环、异常处理 | 三层解释 + 风险提示 |
| 安全、权限、支付、文件操作、网络请求 | 三层解释 + 强风险提示 + 审阅建议 |
| 样式、配置、重复模板代码 | 默认简略，可按需展开 |

### 6.2 解释质量反馈闭环

CodeReader 的解释质量不应建立在“AI 自己给自己打分”上。更合理的闭环是：

```text
AI 生成初始解释
  ↓
用户阅读、标记、质疑
  ↓
系统保存反馈和阅读状态
  ↓
必要时局部重新解释
  ↓
解释层随项目和用户理解持续演进
```

MVP 中至少应提供：

- “有帮助”反馈；
- “不对劲”反馈；
- “用更白话解释”；
- “用更详细解释”；
- “重新解释当前行/代码块”；
- 用户补充笔记。

`confidence: 0.82` 这类数字可以作为内部信息保存，但面向目标用户时应转换为更可读的提示，例如“解释相对明确”“这里依赖上下文较多，建议结合调用处一起看”“建议重点检查”。

## 7. 持久化解释层

### 7.1 为什么必须持久化

如果每次打开代码都重新生成解释，CodeReader 就会退化为普通 AI 问答工具。持久化解释层是产品成立的关键。

持久化意味着：

- 第一次分析后，解释被保存；
- 再次打开同一文件时直接加载已有解释；
- 用户可以修改、补充、确认解释；
- 解释可以有版本；
- 代码变化后可以判断哪些解释仍然有效，哪些需要更新；
- 解释成为项目的一部分认知资产。

### 7.2 解释层与代码的关系

解释层不直接写入源代码文件，而是作为旁路数据保存。

示例：

```text
project/
  src/app.ts
  src/login.ts
  .codereader/
    index.db
    explanation-cache/
    project-map.json
```

解释数据与代码通过以下信息建立映射：

- 文件路径；
- 文件内容 hash；
- 行号范围；
- AST 节点 ID；
- 函数/类/变量符号；
- 代码片段 hash；
- 解释版本号。

## 8. 核心使用场景

### 8.1 场景一：用户打开 AI 生成项目

1. 用户用 CodeReader 打开本地项目；
2. 系统扫描项目结构；
3. 系统识别主要语言、入口文件、模块结构；
4. 系统展示推荐阅读路径；
5. 用户选择一个推荐文件；
6. 系统生成文件概览、函数/代码块解释和用户点击后的关键行解释；
7. 用户逐行阅读并标记理解状态。

### 8.2 场景二：逐行审阅一个关键文件

1. 用户打开 `auth.ts`；
2. 点击第 28 行；
3. 右侧展示该行代码层意义、局部组成意义、上下游关系和风险；
4. 用户点击“查看依赖行”；
5. 代码区高亮相关行；
6. 用户标记该行“已理解”或“有疑问”。

### 8.3 场景三：代码变更后解释层增量更新

1. 用户修改或重新导入 AI 生成的新版本文件；
2. CodeReader 检测到代码 diff；
3. 系统展示变更摘要，例如“新增 3 个函数、修改 12 行、删除 1 个代码块”；
4. 系统判断哪些解释仍然有效，哪些解释过期；
5. 用户只重新生成受影响部分解释；
6. 旧解释保留为历史版本。

### 8.4 场景四：学习真实项目

1. 用户导入一个开源项目或 AI 生成项目；
2. CodeReader 生成推荐阅读路径；
3. 用户先读项目概览，再读入口文件，再读关键函数；
4. 关键行解释带前置知识和学习提示；
5. 用户逐步形成项目认知地图。

### 8.5 场景五：第一次体验 CodeReader

1. 用户首次启动 CodeReader；
2. 启动页展示“打开项目”和“体验示例项目”两个入口；
3. 用户选择入门示例；
4. 左侧展示示例代码，中间展示 Monaco 代码阅读区，右侧展示预置解释层；
5. 用户点击单行、多行、函数，理解解释映射关系；
6. 用户尝试标记“已理解 / 有疑问 / 不对劲”。

该体验必须不依赖用户首次配置 LLM API Key。MVP 不需要复杂新手教程，但需要让用户第一分钟内看到 CodeReader 与普通聊天解释工具的差异。

## 9. 关键界面模块

| 模块 | 说明 | MVP 是否需要 |
|---|---|---|
| 项目打开器 | 打开本地文件夹或单文件 | 必须 |
| 文件树 | 显示项目文件结构 | 必须 |
| 代码阅读区 | 行号、语法高亮、选中行 | 必须 |
| 解释面板 | 展示稳定结构化解释 | 必须 |
| 分析任务面板 | 展示分析进度和错误 | 必须 |
| 解释状态标记 | 已解释/过期/风险/已理解 | 必须 |
| 示例项目体验区 | 内置示例代码和预置解释层 | 必须 |
| 推荐阅读路径 | 推荐先读哪些文件、为什么读 | 必须 |
| 阅读进度 | 展示已读、已理解、有疑问的比例 | MVP 简化 |
| 变更摘要视图 | 展示新增、修改、删除及受影响解释 | 必须 |
| 解释反馈入口 | 有帮助、不对劲、白话/详细、重新解释 | 必须 |
| 搜索 | 搜索文件、函数、解释内容 | MVP 可简化 |
| 项目地图 | 展示文件类型、入口、模块关系摘要 | MVP 文本化，不做完整知识图谱 |
| 聊天追问 | 对某行追问，但不作为主入口 | P1，MVP 不做默认入口 |
| VS Code 插件 | 从 VS Code 打开 CodeReader | 非 MVP |

## 10. 产品边界

### 10.1 MVP 必须守住的边界

- 独立桌面应用；
- 能打开本地代码文件/项目；
- 能以 IDE 式界面展示代码；
- 能生成并保存解释层；
- 能在右侧稳定展示对应解释；
- 能处理代码变化后的解释过期状态。

### 10.2 暂不追求

- 完整替代 VS Code/PyCharm；
- 支持所有编程语言；
- 强大的自动修复能力；
- 团队协作；
- 云端知识库；
- PR 审查平台；
- 复杂架构图自动生成。

---

## 11. 成功指标

### 11.1 用户体验指标

- 用户打开一个 AI 生成文件后，可以在 1 分钟内看到文件级解释；
- 用户点击任意已解释行，右侧能即时展示对应解释；
- 用户再次打开同一文件，解释层直接恢复；
- 用户能明确判断哪些行已理解、哪些行存在风险；
- 用户能通过解释理解关键文件的主要工作方式。

### 11.2 产品验证指标

| 指标 | MVP 目标 |
|---|---|
| 单文件解释成功率 | ≥ 90% |
| 行级解释映射准确率 | ≥ 85% |
| 再次打开解释恢复成功率 | ≥ 95% |
| 局部修改后的过期识别准确率 | ≥ 80% |
| 用户完成关键文件阅读率 | ≥ 60% |

## 12. 产品命名与表达

产品名：**CodeReader**

推荐中文表达：

- AI 代码阅读器；
- AI 代码认知 IDE；
- 面向 AI 编程的代码解释层；
- 帮助用户真正掌握 AI 生成代码的阅读工具。

不建议使用的表达：

- AI 聊天代码助手；
- 代码问答机器人；
- VS Code 插件；
- 代码审查工具；
- 代码注释生成器。

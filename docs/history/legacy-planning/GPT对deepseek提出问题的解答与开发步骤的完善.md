我建议把这些补充吸收进 CodeReader 的产品定义里，但做几处微调。

# 一、对你 7 点想法的判断与补充

## 1. 关于解释质量：不要陷入“AI 检查 AI”的循环

你的判断是对的。

CodeReader 不应该把核心价值建立在：

> AI 生成代码 → AI 解释代码 → AI 检查解释 → AI 再检查检查结果

这种循环上。

这会导致：

- 成本上升；
- 延迟增加；
- 解释结果越来越“自洽”，但不一定更真实；
- 用户仍然没有获得判断力。

所以 CodeReader 的核心机制应该是：

> **AI 负责生成初始解释，人负责确认、质疑、标记和修正。**

我建议把解释质量机制设计成这几个层次：

机制

作用

MVP 是否需要

AI 初始解释

生成第一版解释层

必须

用户反馈：有用 / 不对劲 / 重新解释

让用户参与审阅

必须

用户理解状态：未读 / 已读 / 已理解 / 有疑问

建立阅读进度

必须

局部重新解释

针对某一行、某个代码块重新生成

必须

多模型交叉验证

用另一个模型复核解释

暂缓

自动置信度评分

给出 `confidence`

MVP 不建议作为核心

解释版本历史

保留用户修正与 AI 重解释记录

P1

尤其是 `confidence: 0.82` 这种东西，对目标用户未必有意义。非科班用户看到一个数字，并不能判断它是否可信。

我建议改成更人性化的表达：

不建议

建议

confidence: 0.82

解释可信度：中等

risk\_score: 0.64

建议重点检查

low confidence

这里依赖上下文较多，建议结合调用处一起看

high confidence

这是常见写法，解释相对明确

也就是说，**不要把 AI 置信度伪装成数学准确性，而是把它转化成阅读提示。**

## 2. 关于解释维度：三层解释足够

你这个调整非常关键。

之前设计“表面意义、实际意义、关联意义、全局作用、风险、学习提示……”确实有点过重。对每一行都这样展开，会把用户淹没。

我建议 CodeReader 的核心解释结构就固定为三层：

层级

名称

解释

第一层

**代码层意义**

从语法、代码规则、写法本身解释这一行在做什么

第二层

**局部组成意义**

它在当前函数、代码块或模块中承担什么作用

第三层

**项目全局意义**

它在整个项目流程、业务目标或架构中的位置

可以理解成：

> **这行代码写了什么 → 它在这段逻辑里干什么 → 它对整个项目有什么意义。**

同时，不是每一行都必须三层全开。

我建议采用“默认简洁，按需展开”的设计：

代码类型

默认解释程度

import / 常量 / 简单赋值

简短解释

函数定义 / 类定义 / API 路由 / 数据库操作

三层解释

条件判断 / 循环 / 异常处理

三层解释 + 风险提示

安全、权限、支付、文件操作、网络请求

三层解释 + 强风险提示

样式、配置、重复模板代码

可折叠或批量解释

这样会更符合“读懂并理解是终极目标”，而不是为了逐行解释而逐行解释。

## 3. 关于用户引导：MVP 可以轻量，但不能完全没有

你说可以先准备三份示例代码，这个很适合 MVP。

我建议 MVP 的新手入口不要做复杂 onboarding，而是做成：

## **示例项目体验区**

准备三类代码：

示例

行数

目的

入门示例

30-50 行

让用户快速理解 CodeReader 的左右映射体验

中等示例

80-100 行

复杂示例

100-200 行

展示模块拆分、风险提示、阅读状态、局部展开

每个示例都应该包含：

- 左侧代码；
- 右侧解释层；
- 点击行显示解释；
- 点击函数显示函数级解释；

- 局部重新解释。

不建议一开始做复杂的新手教程。你的产品本身就是阅读器，**让用户一打开就看到一个可阅读的代码文件，比弹窗教学更有效。**

## 4. 关于代码变更：必须纳入核心设计

这一点 DeepSeek 提得对，你的判断也对。

因为 CodeReader 的解释是持久化的，所以一旦代码变化，解释层就会面临一个问题：

> 这条解释还对应现在这行代码吗？

因此，代码变更不是附加功能，而是 CodeReader 的基础能力之一。

我建议设计一个：

## **解释层状态机制**

状态

含义

有效

当前解释与代码匹配

可能过期

附近代码发生变化，但这行未直接改变

已过期

这行代码发生变化，解释需要更新

已删除

原代码行已不存在

新增未解释

新增代码还没有解释

同时增加一个：

## **变更摘要视图**

用户重新打开项目时，CodeReader 应该告诉他：

> 自你上次阅读以来，这个文件新增了 3 个函数，修改了 12 行，删除了 1 个代码块，其中 5 条解释可能已过期。

这个体验非常重要，因为它会让用户知道：

- 哪些地方发生了变化；
- 哪些解释还可信；
- 哪些地方需要重新阅读；
- 哪些 AI 新写的代码还没被自己理解。

Git 集成可以先做轻量：

阶段

Git 能力

MVP

检测文件内容 hash 变化，标记解释过期

P1

读取 git diff，展示变更前后

P2

支持 commit 级解释、PR 阅读、变更影响分析

P3

团队审阅流、解释随 commit 版本流转

## 5. 关于上下文窗口管理：必须分级解释，不能无脑逐行

你的判断非常正确。

CodeReader 不能假设所有代码都适合一次性逐行解释。哪怕模型上下文很大，也不应该把几千行代码全部塞进去逐行分析。

原因有三个：

1. **成本太高**
2. **错误率会上升**
3. **用户也读不完**

所以 CodeReader 应该采用：

> **先模块级理解，再按需行级展开。**

我建议设计一个“解释深度策略”：

文件规模

默认策略

0-200 行

可直接生成文件级 + 函数级 + 关键行级解释

200-800 行

先生成文件级结构和函数级解释，用户点击后展开行级

800-2000 行

先做模块/函数索引，只解释关键入口、调用链和风险区域

2000 行以上

不建议逐行解释；先拆分模块、识别职责、推荐阅读路径

同时，Context Builder 应该不是“把整个文件发给 LLM”，而是根据目标动态构造上下文：

解释目标

上下文组成

单行解释

当前行 + 所在函数 + 相关变量定义 + 调用处摘要

代码块解释

代码块 + 所在函数 + 输入输出变量

函数解释

函数体 + 被调用函数摘要 + 调用该函数的位置

文件解释

文件结构 + 导入导出 + 核心函数摘要

项目解释

文件树 + 入口文件 + 模块摘要 + 依赖关系

也就是说，CodeReader 的核心技术之一应该叫：

> **Context Builder：面向代码解释任务的上下文构造器。**

这会成为产品壁垒。

***

## 6. 项目级认知地图：先做轻量，不要一开始做大图谱

项目级认知地图很重要，但 MVP 不要做复杂知识图谱。

我建议先做三个轻量能力：

### 第一，项目文件地图

告诉用户：

- 项目有哪些目录；
- 哪些是入口；
- 哪些是配置；
- 哪些是业务逻辑；
- 哪些是样式；
- 哪些是数据层。

### 第二，推荐阅读路径

比如：

```
建议阅读顺序：
1. package.json：理解项目依赖和启动方式
2. src/main.tsx：理解应用入口
3. src/App.tsx：理解主界面结构
4. src/api/user.ts：理解用户数据来源
5. src/components/UserPanel.tsx：理解用户数据如何展示

```

### 第三，模块关系摘要

例如：

```
用户登录流程：
LoginPage.tsx 负责收集账号密码
authApi.ts 负责向后端发送登录请求
authStore.ts 保存登录状态
router.ts 根据登录状态控制页面访问

```

这些就已经足够帮助目标用户建立“项目认知地图”。

完整图谱、节点边关系、可视化 DAG 可以放到 P1/P2。

# CodeReader MVP 开发任务说明

## 一、项目定位

CodeReader 是一个独立桌面端 AI 代码阅读 IDE，不是 VS Code 插件。

它的目标用户是：借助 AI 生成代码、但没有系统编程背景，仍然希望真正理解、审阅并掌握自己项目代码的人。

CodeReader 的核心体验是：

- 左侧/中间展示清晰分行的代码；
- 右侧展示与代码行、代码块、函数、文件稳定映射的解释层；
- 解释不是一次性聊天回复，而是本地持久化的认知资产；
- 用户可以对解释进行标记、反馈、重新解释；
- 当代码变更时，系统能标记解释是否过期。

## 二、MVP 技术栈

使用：

- Tauri
- React
- TypeScript
- Monaco Editor
- SQLite
- Tree-sitter
- LLM API

## 三、MVP 开发顺序

### Task 1：创建项目骨架

创建 Tauri + React + TypeScript 桌面应用。\
实现基础三栏布局：

- 左侧：文件树 / 示例入口
- 中间：代码阅读区
- 右侧：解释面板

### Task 2：集成 Monaco Editor

在中间区域集成 Monaco Editor。\
要求：

- 只读展示代码；
- 支持行号；
- 支持语法高亮；
- 支持点击行；
- 点击行后右侧能显示当前行号。

### Task 3：实现本地文件打开

使用 Tauri 文件系统能力打开本地代码文件。\
要求：

- 支持打开单个文件；
- 自动识别语言；
- 读取文件内容；
- 在 Monaco Editor 中展示。

### Task 4：定义解释层数据结构

定义 TypeScript 类型和 SQLite schema。\
核心对象包括：

- project
- file
- code\_snapshot
- code\_symbol
- explanation
- explanation\_target
- explanation\_feedback
- reading\_state
- change\_record

解释目标类型包括：

- line
- range
- block
- function
- class
- file
- module
- project

### Task 5：实现解释层持久化

将解释数据保存到 SQLite。\
要求：

- 解释绑定到文件路径、文件 hash、代码行或代码结构节点；
- 关闭应用后重新打开，解释仍能加载；
- 同一个文件不要每次重新生成解释。

### Task 6：实现右侧解释面板

点击代码行、代码块或函数时，右侧显示对应解释。

解释内容采用三层结构：

1. 代码层意义：从代码规则和写法上解释它做了什么；
2. 局部组成意义：解释它在当前函数、代码块或模块中的作用；
3. 项目全局意义：解释它在整个项目中的作用。

同时支持：

- 已读；
- 已理解；
- 有疑问；
- 不对劲；
- 重新解释。

### Task 7：接入 Tree-sitter

接入 Tree-sitter 解析代码结构。\
MVP 优先支持 TypeScript / JavaScript。

要求识别：

- import
- export
- function
- class
- if / for / try 等代码块
- 函数起止行
- 代码块起止行

并建立行号到函数、代码块、文件的映射。

### Task 8：接入 LLM 生成解释

根据当前文件和 Tree-sitter 解析结果构造上下文，调用 LLM 生成解释 JSON。

解释生成策略：

- 小文件可以生成文件级、函数级、关键行级解释；
- 中等文件先生成文件级和函数级解释，用户点击后再展开行级；
- 大文件不默认逐行解释，先生成模块级和函数级解释。

LLM 输出必须是结构化 JSON，保存到 SQLite 后再展示。

### Task 9：实现代码变更检测

检测文件内容变化。\
要求：

- 计算文件 hash；
- 识别新增、删除、修改；
- 标记解释状态：
  - 有效
  - 可能过期
  - 已过期
  - 新增未解释
  - 已删除
- 提供变更摘要视图。

### Task 10：内置示例与 MVP 打包

准备三个内置示例代码：

- 30-50 行；
- 80-100 行；
- 100-200 行。

每个示例都应该自带解释层，用于用户快速体验。

最终打包为可运行的桌面应用。

# 五、我的最终建议

开发顺序我建议就按这个原则走：

> **先验证阅读体验，再做技术完整性；先稳定解释层，再做智能增强；先单文件闭环，再扩展到项目级认知地图。**

最小闭环不是“AI 能解释代码”，而是：

> 用户打开一个文件 → 看到代码 → 点击某行 → 看到稳定解释 → 标记理解状态 → 关闭再打开仍然存在。

只要这个闭环成立，CodeReader 就和普通 ChatGPT 问答式解释工具拉开差距了。

# CodeReader MVP 开发任务 Prompt v0.4

## 使用说明

本 Prompt 只用于指导 MVP 级别实现，不用于压缩 CodeReader 的全量产品目标。跨文件认知地图、多语言支持、插件入口、团队协作、云同步、PR 审阅和复杂 Git 工作流可以保留在后续路线中，但不得作为 MVP 完成前置条件。

## 项目定位

CodeReader 是一个独立桌面端 AI 代码阅读 IDE，不是 VS Code 插件、不是聊天机器人、不是代码审查插件、不是自动代码生成工具。

MVP 要验证的核心闭环是：

```text
打开代码
  -> 看见结构
  -> 点击行/块/函数
  -> 右侧看到稳定解释
  -> 用户标记已理解/有疑问/不对劲
  -> 解释保存到本地
  -> 代码变更后提示过期
```

主体验必须是：

```text
代码文件 -> 行/块/函数锚点 -> 右侧解释层 -> 用户理解状态 -> 本地持久化
```

不得把 MVP 做成：

```text
用户输入问题 -> AI 回答一段文字
```

## 技术栈

- Tauri
- React
- TypeScript
- Monaco Editor
- SQLite
- Tree-sitter
- LLM API

## MVP 硬约束

1. 必须是独立桌面端应用，VS Code 插件不进入 MVP。
2. 必须先跑通单文件阅读闭环，再扩展项目级能力。
3. 必须支持三栏界面：文件树/示例入口、代码阅读区、解释面板。
4. 必须支持单行点击、多行选择、函数/代码块解释切换。
5. 必须先用 Tree-sitter 识别函数、类、代码块边界，再做行级按需解释。
6. 解释必须保存到 SQLite，再次打开不能自动重新生成。
7. 解释不能只绑定行号，必须绑定文件快照、结构节点、行范围、代码 hash 和锚点文本。
8. 用户必须能标记“已理解 / 有疑问 / 不对劲”，并能写备注。
9. 代码变化后必须能标记 `valid / stale / invalid / new_unexplained / deleted`。
10. 首次体验必须不依赖 API Key，内置示例和预置解释层要可用。

## 解释展示原则

右侧面板默认展示三层核心解释：

1. 代码层意义；
2. 局部组成意义；
3. 项目全局意义。

以下内容默认折叠或作为辅助区域展示：

- 风险；
- 前置知识；
- 审阅建议；
- 相关行；
- 可信提示；
- 用户笔记；
- 解释版本。

不得在用户界面把 `confidence: 0.82` 作为主要判断依据。需要表达不确定性时，使用白话提示，例如“这里依赖上下文较多，建议结合调用处一起看”。

## MVP 任务顺序

### Task 0：纯网页体验原型

目标：验证“代码锚点 + 右侧解释层”是否成立。

要求：

- Monaco 展示预置 JS/TS 示例代码；
- 右侧展示预置解释 JSON；
- 支持点击单行；
- 支持多行选择；
- 支持函数级解释切换；
- 支持“已理解 / 有疑问 / 不对劲”反馈；
- 不配置 API Key 也能完整体验。

### Task 1：Tauri + React 基础应用

目标：创建独立桌面端三栏应用。

输出：

- 左侧：文件树 / 示例入口；
- 中间：只读代码阅读区；
- 右侧：解释面板；
- 底部：状态栏。

### Task 2：Monaco Editor 集成

要求：

- 只读展示代码；
- 显示行号；
- 支持语法高亮；
- 支持点击行；
- 支持多行选择；
- 当前选择同步到右侧面板。

### Task 3：本地文件和项目打开

要求：

- 打开单个文件；
- 打开本地项目文件夹；
- 忽略 `node_modules`、`.git`、`dist`、`build`；
- 以真实目录层级展示项目结构，支持目录展开/收起、文件选中态和加载态；
- 同名文件必须通过所在目录清晰区分，前后端使用稳定的目录/文件节点数据，不依赖脆弱的路径字符串临时拼树；
- 自动识别 JS/TS/JSX/TSX 文件，并允许它们进入 Tree-sitter 与结构化解释链路；
- README、Markdown、JSON、YAML、TOML 和常见配置文件至少可以只读预览，并明确标注“暂不支持结构化代码解释”；
- 二进制、超大文件和符号链接不得被当作文本误读，必须给出明确的不可预览原因；
- 保持目录遍历深度、条目数量、文件大小、安全路径和符号链接边界。

边界说明：

- “项目树看见更多文件”不等于“基础 MVP 解释更多语言”；
- 基础 MVP 的结构化解释仍优先验证 JS/TS/JSX/TSX；
- Python、SQL 等语言应在基础闭环稳定后，作为独立语言扩展逐项接入和验收，不得用正则或伪 AST 绕过正式解析器。

已确认的语言扩展顺序：

1. 基础 MVP：先以 JS/TS/JSX/TSX 验证单文件解释、持久化和变更检测闭环；
2. 第一扩展语言：Python，必须完整接入文件识别、Monaco、Tree-sitter、四类 Context Builder、LLM、SQLite 和变更检测；
3. 第二扩展语言：SQL，必须在 Python 独立验收后另建分支推进，并明确方言边界、语句结构、数据流语义和上下文预算；
4. 每种新增语言都必须独立测试、桌面验收和 Git 提交，不能只做到“能打开”或“有语法高亮”。

当前扩展实现收束：

- Python 已作为第一扩展语言完成完整解释链路验证；
- SQL 以通用 SQL 及常见 PostgreSQL、MySQL、SQLite 语法交集为首阶段边界，不承诺覆盖全部厂商方言；
- SQL 结构目标包括文件、顶层语句、CTE/子查询、关键数据流子句和任意单行/多行选择；
- SQL Context Builder 必须提取读取关系、写入目标、CTE、函数调用、过滤、连接、聚合、排序和限制线索，不连接数据库时必须明确说明缺少 schema 与运行时查询计划；
- SQL 文件级上下文只发送结构目录、语句入口和 CTE 摘要，不把大型脚本整体发送给模型；
- Python、SQL 均复用现有结构化 JSON 协议、API Key 安全存储、SQLite 恢复、失败不覆盖和变更检测机制。

### Task 4：Tree-sitter 结构解析

要求识别：

- import/export；
- function/class；
- if/for/try/catch；
- SQL statement、CTE、subquery、JOIN、WHERE、GROUP BY、ORDER BY、LIMIT 等主要结构；
- 函数和代码块起止行；
- 行号到函数/代码块/文件的映射。

### Task 5：SQLite 和解释层 schema

至少实现：

- projects；
- files；
- code_snapshots；
- code_nodes；
- explanation_nodes；
- explanation_targets；
- explanation_feedback；
- user_reading_states；
- change_records；
- reading_paths。

解释目标至少保存：

```text
project_id
file_id
file_path
file_hash
snapshot_id
target_type
start_line
end_line
symbol_id
code_hash
anchor_text
status
created_at
updated_at
```

### Task 6：mock 解释层持久化

要求：

- 将预置解释保存到 SQLite；
- 根据文件路径、快照、代码 hash、行/范围/结构节点加载解释；
- 关闭重开后解释仍存在；
- 用户阅读状态和反馈能恢复。

### Task 7：解释面板

要求：

- 默认展示三层解释；
- 支持白话/详细模式；
- 风险、前置知识、审阅建议、相关行折叠展示；
- 支持有帮助、不对劲、重新解释和用户备注。

### Task 8：Context Builder

根据解释目标构造上下文，不把大型文件直接整体发送给模型。

| 目标 | 上下文 |
|---|---|
| 单行 | 当前行 + 所在函数 + 相关变量定义 |
| 多行/代码块 | 代码块 + 所在函数 + 输入输出变量 |
| 函数 | 函数体 + 相关 import + 被调用函数摘要 |
| 文件 | 文件结构 + import/export + 核心函数摘要 |

### Task 9：LLM 解释生成

要求：

- 输出结构化 JSON；
- JSON 校验失败不写入数据库；
- 支持文件级、函数级、代码块级、关键行解释；
- 支持局部重新解释；
- 支持失败重试和错误提示。

### Task 10：代码变更检测

要求：

- 计算文件 hash；
- 重新解析 AST；
- 匹配旧节点和新节点；
- 迁移可复用解释；
- 标记过期或新增未解释；
- 生成变更摘要；
- 只重新解释受影响区域。

### Task 11：第一公里引导

MVP 只做轻量项目地图，不做完整知识图谱。

要求：

- 内置 3 个示例代码文件；
- 识别入口文件、配置文件、业务文件；
- 生成推荐阅读路径；
- 展示阅读进度。

### Task 12：MVP 打包与验收

MVP 通过标准：

- 独立桌面端可运行；
- 示例体验可用且无需 API Key；
- 单文件解释闭环可用；
- 持久化恢复可用；
- 单行、多行、函数解释映射可用；
- 反馈和阅读状态可保存；
- 变更摘要和过期标记可用；
- 受影响区域可局部重新解释。

# CodeReader 版本沿革与兼容性说明

[English product information](../../README.md) | [中文使用说明](../../README.zh-CN.md) | [完整变更日志](../../CHANGELOG.md)

本文面向用户、测试人员和发布维护者，说明 CodeReader 从早期原型到 1.0 的功能演进，以及升级时必须遵守的兼容性边界。旧的 MVP、内测、规划记录仍保留在本目录下，供追溯使用；它们不是当前产品行为的权威来源。

## 0.10.x 到 0.11.x：阅读工作流建立

0.10.x 引入真实目录层级、安全文本预览、阅读路径和文件级进度，并扩展 Python、SQL 的结构化解释。它还建立了 Windows x64 的早期 NSIS/MSI 包装和恢复验证。

0.11.x 内测阶段没有改变“本地优先、用户选择文件后才访问”的方向，而是把这一方向变成可维护的系统能力：

- 迁移使用 SQLite 版本控制与事务边界。
- 模型调用、解释上下文、错误分类和前端工作流被拆分为可测试的模块。
- 提示词支持登记、灰度和回滚，已有模型配置与凭据引用是兼容性义务。
- 阅读进度、反馈和诊断在保留隐私边界的前提下持续完善。

详细的 beta 验收、真实项目验证和历史设计资料保留在本目录的 beta 子目录。

## 1.0.0-rc.1：从内测能力到生产候选

1.0 候选版将项目定位从 MVP/内测转为受维护的本地桌面产品：

- 文件或目录只能由原生选择器授予，界面层不获得任意路径或网络权限。
- 用户可以打开任意有权限的位置；树中保留普通文件，主阅读区只渲染可安全支持的格式。
- Markdown 成为一等阅读格式；图片预览、文本/代码预览都具有边界；不支持、二进制或过大文件显示元数据而不替换当前文档。
- 后台扫描、刷新与解释生成绑定到目标身份，旧任务不能覆盖当前文件状态。
- 解释、进度、提示词版本、模型配置和凭据引用延续 0.10.x/0.11.x 的兼容义务。
- 数据库升级先验证备份，再在单一事务中迁移并进行完整性检查；无法安全恢复时进入非破坏性恢复状态。
- 发布矩阵覆盖 Windows/Linux 的 x64、ARM64，生成十个安装包以及校验和、SPDX SBOM、artifact attestations 和原生 smoke 证据。

## 升级与数据兼容性

### 受支持的升级来源

- 0.10.x：CodeReader 会发现受支持的旧应用数据位置，只有在当前数据库不存在时才导入；绝不会用旧数据库覆盖已存在的当前数据。
- 0.11.x：沿用已保存的解释、阅读进度、提示词版本、模型配置与凭据引用，并通过当前 schema 迁移升级。

### 每次迁移的安全流程

1. 打开旧数据库并执行完整性检查。
2. 创建且验证备份。
3. 在单一事务中应用所需迁移。
4. 执行 schema、外键和数据完整性验证。
5. 成功后打开新数据库；失败时保留原始数据库和恢复路径。

### 不支持的操作

- 不支持在没有备份的情况下将新数据库降级给更早版本。
- 不支持为了跳过恢复状态而手动删除数据库或备份。
- 不承诺对未记录的开发快照、手工篡改数据库或未知未来 schema 的兼容性。

出现恢复状态时，请保留数据库与备份，记录应用版本、系统、稳定错误码和复现条件，再提交脱敏后的 Issue。

## 发布通道与版本号

- 形如 1.0.0-rc.N 的版本是候选版，用于完整生产验证。它可能在发现问题后被后续候选版替代。
- 1.0.0 是稳定版。稳定版不应覆盖或静默替换已发布资产。
- Windows/Linux 首发支持 x64 与 ARM64；macOS 在下一版本规划中。

版本号与安装包、package.json、Cargo、Tauri 配置和 Windows MSI 数值版本必须一致。发布时应以 GitHub Release 页面和随附 SHA256SUMS、SBOM、attestation 为准。

# CodeReader MVP 实现标准

版本：v0.1  
日期：2026-06-08  
项目名：CodeReader  
文档性质：MVP 范围、验收标准与开发计划

---

## 1. MVP 目标

CodeReader MVP 的目标不是做一个完整 IDE，也不是做一个全功能代码 agent，而是验证一个核心命题：

> 用户能否在 IDE 式界面中打开一个 AI 生成的代码文件，并逐行看到稳定保存的解释，从而理解代码的表面意义、实际意义、上下文关系和潜在风险。

MVP 必须证明以下体验成立：

1. 代码是主体。
2. 解释在右侧稳定映射。
3. 解释可以保存和重新打开。
4. 用户可以逐行阅读并标记理解状态。
5. 代码修改后，解释不会无差别丢失，而是能识别过期范围。

---

## 2. MVP 产品形态

MVP 推荐使用 VS Code Extension 实现。

原因：

- 用户已经熟悉 VS Code 类 IDE 体验。
- VS Code 原生支持代码展示、行号、语法高亮、文件树、文件监听。
- 可以将主要精力放在解释层、映射关系、持久化和右侧面板。

MVP UI：

```text
VS Code
├─ Explorer 文件树
├─ Editor 代码阅读区
│  ├─ 行号
│  ├─ 语法高亮
│  ├─ 行级解释状态 gutter 标记
│  └─ 风险标记
└─ CodeReader Sidebar / Webview
   ├─ 当前文件概览
   ├─ 当前行解释
   ├─ 关系说明
   ├─ 风险提示
   ├─ 学习提示
   └─ 理解状态按钮
```

## 3. MVP 必须实现的功能

### 3.1 项目与文件

- 打开当前 VS Code workspace。
- 展示当前文件分析状态。
- 支持分析当前打开文件。
- 支持 JS / TS / Python 至少一种语言，建议优先 TypeScript / JavaScript。
- 默认忽略 `node_modules`、`.git`、`dist`、`build`。

### 3.2 代码显示与行映射

- 使用 VS Code 原生编辑区。
- 点击代码行后，右侧面板显示该行解释。
- 已解释行在 gutter 或 decoration 中显示状态。
- 有风险行显示风险标记。

### 3.3 解释生成

必须生成以下内容：

#### 文件级解释

- 文件职责。
- 主要导入。
- 主要导出。
- 核心函数列表。
- 文件在项目中的可能作用。

#### 函数级解释

- 函数职责。
- 参数说明。
- 返回值说明。
- 内部主要流程。
- 副作用。
- 风险。

#### 行级解释

每行至少包含：

- 表面意义。
- 实际意义。
- 上下文关系。
- 全局作用。
- 潜在风险。

### 3.4 持久化

- 使用 SQLite 保存解释。
- 关闭 VS Code 后再打开，解释仍可加载。
- 保存每条解释对应的文件路径、行号、代码哈希、解释内容、生成时间、模型名称。
- 保存用户理解状态。

### 3.5 代码变更检测

- 监听当前文件保存事件。
- 文件变化后重新计算文件哈希。
- 如果某行代码哈希变化，将对应解释标记为 stale。
- 用户可以点击“更新当前文件解释”。

### 3.6 用户状态

用户可对当前行标记：

- 未读。
- 已读。
- 已理解。
- 不理解。
- 需复查。

状态必须保存并可恢复。

### 3.7 风险提示

MVP 至少识别并展示以下风险：

- 可能空值访问。
- 异常处理缺失。
- 硬编码密钥或敏感信息。
- 未校验用户输入。
- 明显不理解上下文的低置信度解释。

## 4. MVP 明确不做的功能

为避免范围失控，MVP 不做：

- 不做完整独立 IDE。
- 不做多人协作。
- 不做云同步。
- 不做完整项目级知识图谱。
- 不做自动修改代码。
- 不做自动提交 Git。
- 不做所有语言支持。
- 不做复杂权限系统。
- 不做完整商业计费系统。
- 不做复杂可视化图谱。

## 5. MVP 用户流程

### 5.1 首次分析流程

```text
用户打开 VS Code 项目
  ↓
打开一个代码文件
  ↓
点击 CodeReader: Analyze Current File
  ↓
系统读取当前文件
  ↓
系统解析文件结构
  ↓
系统调用模型生成解释
  ↓
解释写入 SQLite
  ↓
代码行显示解释状态
  ↓
用户点击任意行查看右侧解释
```

### 5.2 再次打开流程

```text
用户重新打开项目
  ↓
打开已分析文件
  ↓
CodeReader 检查本地解释库
  ↓
加载已有解释
  ↓
用户继续阅读，无需重新生成
```

### 5.3 文件修改流程

```text
用户修改并保存代码
  ↓
CodeReader 重新计算代码哈希
  ↓
比对已有解释锚点
  ↓
未变化解释保持 valid
  ↓
变化区域标记 stale
  ↓
用户选择是否重新生成 stale 区域解释
```

## 7. MVP 验收标准

### 7.1 核心体验验收

MVP 通过的最低标准：

1. 用户能打开一个本地项目。
2. 用户能分析当前代码文件。
3. 系统能生成文件级、函数级、行级解释。
4. 用户点击任意已解释代码行，右侧能看到对应解释。
5. 关闭并重新打开后，解释仍存在。
6. 修改代码后，旧解释能被标记为可能过期。
7. 用户能标记某行“已理解”。
8. 风险行能被标记并在右侧说明。

### 7.2 质量验收

- 解释内容不得只有语法翻译，必须结合当前文件上下文。
- 风险提示必须说明原因。
- 低置信度内容必须明确标记。
- 解释结果必须符合结构化 schema。
- 生成失败不得破坏已有解释。

### 7.3 性能验收

建议标准：

- 300 行以内文件能够完整生成解释。
- 解释生成时 UI 不应冻结。
- 再次打开已分析文件时，解释加载应明显快于重新生成。
- 单次文件分析失败可重试。

## 8. 开发阶段拆分

### 8.1 Sprint 0：原型验证

目标：验证 VS Code 中点击行并显示右侧解释。

任务：

- 创建 VS Code Extension。
- 读取当前文件。
- 创建 Webview 解释面板。
- 点击行同步行号。
- 使用 mock 数据显示解释。

产出：

- 可点击行。
- 可显示右侧解释。
- 无需真实 AI。

### 8.2 Sprint 1：解释生成

目标：接入模型并生成结构化解释。

任务：

- 设计 JSON schema。
- 编写 prompt。
- 调用模型 API。
- 校验输出。
- 展示真实解释。

产出：

- 当前文件可生成真实解释。

### 8.3 Sprint 2：持久化

目标：解释可保存、可恢复。

任务：

- 集成 SQLite。
- 设计表结构。
- 保存解释。
- 重新打开文件时加载解释。
- 显示解释状态。

产出：

- 稳定解释层成立。

### 8.4 Sprint 3：基础代码结构分析

目标：解释开始理解函数和代码结构。

任务：

- 接入 Tree-sitter 或基础解析器。
- 识别函数范围。
- 按函数组织解释任务。
- 建立行到函数的映射。

产出：

- 函数级解释和行级解释上下文增强。

### 8.5 Sprint 4：变更检测与过期机制

目标：代码修改后解释可维护。

任务：

- 监听文件保存。
- 计算行级代码哈希。
- 标记 stale。
- 支持重新生成 stale 区域。

产出：

- 解释层具备基本生命周期。

### 8.6 Sprint 5：风险与阅读状态

目标：形成完整 MVP 体验。

任务：

- 风险识别和展示。
- 用户理解状态。
- 文件阅读进度。
- Markdown 导出。

产出：

- MVP 可对外演示。

## 9. MVP 示例演示脚本

演示项目：一个由 AI 生成的 Express / FastAPI 小项目。

演示流程：

1. 打开项目。
2. 打开入口文件。
3. 点击“Analyze Current File”。
4. 等待生成解释。
5. 点击 `import` 行，展示导入解释。
6. 点击 app 初始化行，展示全局作用。
7. 点击路由处理函数，展示参数、响应、风险。
8. 展示某行缺少 try/catch 的风险。
9. 标记几行“已理解”。
10. 修改一行代码并保存。
11. 展示解释过期标记。
12. 重新生成过期解释。
13. 关闭并重新打开，展示解释仍然存在。

## 10. MVP 风险控制

### 10.1 控制语言范围

只支持 1-2 种语言，建议：

- TypeScript / JavaScript 优先。
- Python 第二优先。

### 10.2 控制项目范围

只做当前文件分析，不急于全项目自动分析。

### 10.3 控制解释深度

MVP 默认标准解释，不做多种复杂模式。

### 10.4 控制模型选择

先支持一个主流模型供应商，再做适配层扩展。

### 10.5 控制 UI 复杂度

先用 VS Code 原生能力，不做完整自研 IDE。

---

## 11. MVP 成功定义

如果一个目标用户完成以下动作，MVP 就算成功：

1. 用 AI 生成一个代码文件。
2. 打开 CodeReader。
3. 逐行阅读右侧解释。
4. 能说清楚这个文件大概做什么。
5. 能指出至少一个自己以前没注意到的风险。
6. 第二天重新打开还能看到昨天的解释和理解状态。

这说明 CodeReader 已经从“临时解释工具”变成“代码认知层”。

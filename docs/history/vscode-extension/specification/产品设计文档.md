版本：v0.1

项目名：CodeReader  
文档性质：产品定位、产品形态、核心体验与交互设计说明

---

## 1. 项目背景

随着 AI 编程工具的发展，越来越多非科班用户可以通过自然语言表达需求，并由 AI 生成可运行的代码。对这类用户来说，AI 极大降低了“写出代码”的门槛，但没有同步降低“理解代码”的门槛。

这导致一个新的问题：

> 用户能够拥有一个由 AI 生成的项目，却未必真正理解这个项目是如何实现的。

对于文科背景、产品经理、创业者、独立开发者、设计师、运营人员等用户来说，这种情况尤其明显。项目可以运行，但代码逻辑、依赖关系、风险点、异常路径和维护方式仍然是黑箱。

这种黑箱会带来四类问题：

1. **认知风险**：用户无法判断代码是否符合自己的真实意图。
2. **责任风险**：项目属于用户，但用户无法对代码质量和运行后果负责。
3. **维护风险**：一旦出现 bug、接口变更、业务调整，用户只能继续依赖 AI，而不是形成自己的判断。
4. **学习断层**：传统编程学习从语法到项目逐步推进，而 AI 编程用户往往是先得到完整项目，再倒推理解代码。

CodeReader 试图解决的不是“AI 能否写代码”，而是：

> **AI 写完代码之后，人如何重新理解、审阅、掌握并长期维护它。**

## 2. 产品定位

CodeReader 是一个 **IDE 形态的 AI 代码阅读器**。

它不是普通的聊天式代码解释工具，也不是只做 bug 检查的代码审查工具，而是一个围绕代码文件建立的、可持久化的、可映射到行级和结构级的解释层。

### 2.1 一句话定义

> **CodeReader 给 AI 生成的代码加上一层稳定、可导航、可维护的解释层，让用户真正读懂并掌握自己的项目。**

### 2.2 产品关键词

- AI 代码阅读器
- IDE 形态解释层
- 行级代码认知
- 持久化注释层
- AI 代码 ownership layer
- 面向 AI-native builder 的代码理解工具

### 2.3 产品不是

CodeReader 不应该被设计成以下形态：

- 不是“粘贴一段代码，AI 回复一段解释”的网页问答工具。
- 不是普通 AI Chat。
- 不是单纯的代码审查工具。
- 不是只面向专业程序员的静态分析平台。
- 不是自动修代码优先的 agent。

CodeReader 的核心动作不是“替用户继续写代码”，而是“帮助用户读懂已经存在的代码”。

## 3. 产品目的

CodeReader 的产品目的可以分为三层。

### 3.1 第一层：帮助用户看懂代码

让用户能够逐行理解：

- 这一行代码表面上做了什么。
- 它在运行时实际产生什么效果。
- 它依赖前面哪些变量、函数、配置或外部输入。
- 它会影响后续哪些代码。
- 它在当前函数、文件、模块里的作用是什么。

### 3.2 第二层：帮助用户审阅代码

让用户能够识别：

- AI 是否写了不必要的代码。
- 是否存在空值、异常、安全、性能、依赖、权限、并发等风险。
- 某一行代码是否可能导致副作用。
- 当前实现是否和用户原始意图一致。
- 哪些地方需要补测试或人工确认。

### 3.3 第三层：帮助用户沉淀项目认知

CodeReader 不只生成一次性解释，而是将解释保存为稳定的项目认知资产：

- 用户再次打开同一文件时，可以继续看到此前解释。
- 用户可以标记“已理解”“不理解”“需复查”“有风险”。
- 代码变更后，系统只更新受影响解释。
- 随着项目成长，解释层也随项目一起演进。

## 4. 目标用户

### 4.1 核心用户

**AI-native builder**：主要通过 AI 编程工具生成代码，但缺少系统工程训练的人。

典型用户包括：

- 文科生、非科班开发者。
- 使用 Cursor、Claude Code、Codex、Cline、Copilot 等工具搭项目的人。
- 独立开发者和创业者。
- 产品经理、设计师、运营人员。
- 想通过 AI 做应用，但希望真正理解项目的人。

### 4.2 次级用户

- 编程初学者：用于理解开源项目和 AI 生成代码。
- 技术负责人：用于给团队成员生成解释型代码导览。
- 教育场景老师：用于生成结构化教学材料。
- 代码审查人员：用于辅助理解复杂文件。

## 5. 用户核心需求

用户真正需要的不是“解释一段代码”，而是以下能力：

1. **我想知道每一行代码是什么意思。**
2. **我想知道这一行为什么出现在这里。**
3. **我想知道它和前后代码有什么关系。**
4. **我想知道它在整个文件、模块、项目里的作用。**
5. **我想知道它有没有潜在风险。**
6. **我想知道 AI 有没有写出我并不理解、也不应该盲目接受的东西。**
7. **我想以后再打开项目时，仍然能看到稳定的解释，而不是重新问一遍。**
8. **我想通过阅读自己的项目，逐步提升代码理解能力。**

## 6. 产品形态

### 6.1 IDE 式界面

CodeReader 应接近 VS Code / PyCharm 的阅读体验：

```text
┌───────────────────────────────────────────────────────────────┐
│ 顶部：项目名 / 当前文件 / 分析状态 / 搜索 / 同步状态             │
├───────────────┬──────────────────────────┬────────────────────┤
│ 文件树         │ 代码阅读区                 │ 解释映射区          │
│               │ 带行号、语法高亮、折叠       │ 与左侧行/块稳定对应 │
│ src/          │                          │                    │
│ ├─ app.ts     │  1 import ...             │ 当前行解释          │
│ ├─ routes.ts  │  2 const ...              │ 表面意义            │
│ └─ db.ts      │  3 function ...           │ 实际意义            │
│               │                          │ 关系说明            │
│               │                          │ 风险说明            │
├───────────────┴──────────────────────────┴────────────────────┤
│ 底部：问题面板 / 风险面板 / 解释生成日志 / 项目地图               │
└───────────────────────────────────────────────────────────────┘
```

### 6.2 左侧：代码主体

左侧或主区域展示真实代码文件，包含：

- 文件树。
- 行号。
- 语法高亮。
- 函数/类/代码块折叠。
- 当前解释覆盖状态。
- 风险标记。
- 用户理解状态标记。

代码永远是主体，解释是附着在代码上的认知层。

### 6.3 右侧：解释映射区

右侧不是聊天窗口，而是解释面板。解释面板根据左侧选中对象变化：

| 左侧对象 | 右侧解释内容 |
|---|---|
| 单行代码 | 表面意义、实际意义、上下文关系、后续影响、风险 |
| 多行代码块 | 代码块目标、执行顺序、输入输出、异常路径 |
| 函数 | 函数职责、参数、返回值、副作用、调用关系、测试建议 |
| 文件 | 文件职责、核心结构、导入导出、与其他文件关系 |
| 模块 | 模块边界、依赖关系、业务职责、风险聚合 |
| 项目 | 架构地图、数据流、控制流、关键风险、阅读路径 |

### 6.4 解释层必须持久化

CodeReader 的核心不是每次打开重新问 AI，而是生成并保存解释层。

解释层应具备：

- 稳定保存。
- 可重新加载。
- 可人工编辑。
- 可标记理解状态。
- 可版本化。
- 可随代码变更增量更新。
- 可导出为 Markdown / JSON / HTML。

## 7. 解释内容设计

### 7.1 行级解释字段

每一行解释建议包含：

| 字段 | 说明 |
|---|---|
| 表面意义 | 这行代码字面做了什么 |
| 实际意义 | 运行时造成什么效果 |
| 上下文依赖 | 它依赖前面哪些变量、函数、配置、外部输入 |
| 后续影响 | 后面哪些代码会使用它的结果 |
| 全局作用 | 它服务于当前函数/文件/模块的什么目标 |
| 潜在风险 | 空值、异常、安全、性能、边界条件、AI 幻觉 |
| 学习提示 | 非科班用户需要理解的概念 |
| 审阅建议 | 是否应保留、修改、测试、追问 |
| 置信度 | AI 对解释准确性的自评及依据 |
| 状态 | 未读、已读、已理解、需复查、已过期 |

## 8. 主要使用场景

### 8.1 首次接管 AI 生成项目

用户通过 AI 生成了一个项目，但不知道项目结构。CodeReader 应提供：

1. 项目结构导览。
2. 推荐阅读顺序。
3. 每个文件职责说明。
4. 关键函数解释。
5. 逐行解释。
6. 风险聚合。

### 8.2 阅读某个 AI 生成文件

用户打开一个陌生文件，希望逐行理解。

CodeReader 应支持：

- 自动生成文件概览。
- 按函数/代码块折叠解释。
- 点选某行查看解释。
- 高亮这行依赖的上游代码。
- 高亮这行影响的下游代码。
- 标记“已理解”。

### 8.3 审阅 AI 修改后的代码

用户让 AI 修改了若干文件，希望知道改了什么。

CodeReader 应支持：

- diff 阅读模式。
- 新增/删除/修改行解释。
- 解释“这次改动的目的”。
- 判断哪些旧解释过期。
- 重新生成受影响解释。

### 8.4 学习项目中的具体概念

用户看到不懂的概念，例如 hook、middleware、Promise、ORM、dependency injection。

CodeReader 应支持：

- 在解释面板中提供概念卡片。
- 用当前项目里的代码作为例子解释概念。
- 区分“通用编程概念”和“本项目中的具体用法”。

### 8.5 风险排查

用户希望知道项目中哪里可能有问题。

CodeReader 应支持：

- 风险列表。
- 按严重程度排序。
- 跳转到相关代码行。
- 给出测试建议和人工确认建议。

## 9. 核心交互

### 9.1 打开项目

用户打开本地项目或 GitHub 仓库。CodeReader 展示文件树，并提示是否生成解释层。

### 9.2 生成解释层

用户选择分析范围：

- 当前文件。
- 当前目录。
- 全项目。
- 仅 AI 最近修改的文件。

系统生成：

- 文件摘要。
- 函数摘要。
- 行级解释。
- 关系边。
- 风险提示。

### 9.3 阅读解释

用户点击代码行，右侧显示对应解释。用户可以：

- 标记已理解。
- 标记不理解。
- 追加自己的笔记。
- 请求重新解释。
- 查看上游依赖。
- 查看下游影响。

### 9.4 增量更新

当代码发生变化：

- 未变化行解释保持不变。
- 轻微移动的代码行通过锚定机制重新定位。
- 修改过的代码块标记为“解释可能过期”。
- 用户可以只重新生成过期部分。

### 9.5 导出认知资产

用户可以导出：

- 当前文件解释 Markdown。
- 项目阅读报告。
- 风险清单。
- 学习笔记。
- 解释层 JSON。

## 11. 产品边界

### 11.1 MVP 阶段应该做

- 单项目打开。
- 单文件解释。
- 行级解释。
- 函数级解释。
- 解释持久化。
- 代码行与解释面板映射。
- 基础风险提示。
- 基础增量更新。

### 11.2 MVP 阶段不做

- 不做完整代码自动修复 agent。
- 不做多人协作。
- 不做复杂云端权限系统。
- 不做所有语言支持。
- 不做完整生产级静态分析器。
- 不做自动部署。
- 不做替代 VS Code 的完整 IDE。

---

## 12. 成功指标

### 12.1 用户价值指标

- 用户能否在 10 分钟内理解一个陌生文件的主要作用。
- 用户能否准确说出关键函数的输入、输出和风险。
- 用户是否愿意在 AI 生成代码后主动打开 CodeReader 阅读。
- 用户是否形成“已理解代码行”的进度感。

### 12.2 产品指标

- 文件解释生成成功率。
- 行级解释覆盖率。
- 解释重新加载成功率。
- 代码变更后的解释锚定成功率。
- 用户标记“已理解”的比例。
- 用户重新生成解释的频率。
- 风险提示被用户采纳的比例。

---

## 13. 产品核心判断

CodeReader 的核心不是“解释得更聪明”，而是：

> **让解释变成代码旁边稳定存在的结构化层。**

如果解释每次都随机生成，它就是聊天工具。  
如果解释能够稳定映射、持续更新、长期保存，它才是 CodeReader。

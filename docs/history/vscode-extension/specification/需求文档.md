# CodeReader 需求文档

版本：v0.1  
日期：2026-06-08  
项目名：CodeReader  
文档性质：产品需求规格说明

---

## 1. 需求总述

CodeReader 面向 AI 编程用户，提供 IDE 形态的代码阅读和解释能力。系统应支持对代码文件进行行级、代码块级、函数级、文件级和模块级解释，并将解释结果稳定持久化保存。

系统的基本目标是：

1. 帮助用户理解 AI 生成代码。
2. 帮助用户建立对项目的认知。
3. 帮助用户识别代码潜在风险。
4. 帮助用户将解释信息沉淀为长期可维护资产。

---

## 2. 用户角色

| 角色 | 描述 | 核心需求 |
|---|---|---|
| AI 编程新手 | 通过 AI 写代码但缺少系统编程训练 | 看懂每行代码，知道项目如何运行 |
| 非科班独立开发者 | 能用 AI 搭项目，但维护能力不足 | 掌握项目结构，识别风险，持续学习 |
| 产品/创业者 | 用 AI 实现业务原型 | 判断实现是否符合业务意图 |
| 编程学习者 | 通过真实项目学习代码 | 基于项目上下文理解概念 |
| 技术审阅者 | 帮别人检查 AI 生成代码 | 快速定位风险和关键逻辑 |

## 4. 功能需求

### 4.1 项目打开与文件读取

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-001 | 用户可以打开本地代码项目 | P0 | 可以选择本地目录并展示文件树 |
| FR-002 | 系统可以识别常见代码文件 | P0 | 至少支持 `.js`、`.ts`、`.tsx`、`.py`、`.json`、`.md` 文件展示 |
| FR-003 | 系统可以忽略不应分析的文件 | P0 | 默认忽略 `node_modules`、`.git`、`dist`、`build`、虚拟环境、二进制文件 |
| FR-004 | 用户可以打开单个文件进行阅读 | P0 | 点击文件树中的文件后，主区域展示代码与行号 |
| FR-005 | 系统可以显示文件基础元信息 | P1 | 展示路径、语言、行数、是否已分析、解释版本 |

### 4.2 IDE 式代码阅读界面

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-006 | 代码区应支持行号和语法高亮 | P0 | 用户可以清晰看到带行号的代码 |
| FR-007 | 代码区应支持点击单行 | P0 | 点击某行后右侧展示对应解释 |
| FR-008 | 代码区应支持选中多行 | P1 | 选中多行后右侧展示代码块解释 |
| FR-009 | 代码行应显示解释状态 | P1 | 可看到未解释、已解释、过期、有风险等状态 |
| FR-010 | 风险行应有视觉标记 | P1 | 存在风险的行在 gutter 或行侧显示标识 |

### 4.3 解释生成

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-011 | 系统可以生成文件级概览 | P0 | 对当前文件生成职责、核心流程、主要函数摘要 |
| FR-012 | 系统可以生成函数级解释 | P0 | 对函数生成输入、输出、职责、副作用、调用关系说明 |
| FR-013 | 系统可以生成行级解释 | P0 | 对每个有效代码行生成表面意义和实际意义 |
| FR-014 | 系统可以生成关联意义 | P0 | 对关键行说明上游依赖和下游影响 |
| FR-015 | 系统可以生成风险提示 | P0 | 至少识别空值、异常、权限、安全、性能、边界条件等基础风险 |
| FR-016 | 系统可以生成学习提示 | P1 | 对初学者解释关键概念，如异步、闭包、状态、路由、中间件等 |
| FR-017 | 系统可以生成审阅建议 | P1 | 对风险行给出检查、测试或修改建议 |
| FR-018 | 系统应支持解释深度配置 | P2 | 用户可选择简洁、标准、详细三种解释深度 |

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-019 | 右侧解释面板随代码行切换 | P0 | 点击不同代码行后，解释面板同步更新 |
| FR-020 | 解释面板按结构展示 | P0 | 至少包含表面意义、实际意义、关联意义、风险 |
| FR-021 | 解释面板显示当前解释状态 | P1 | 显示已生成、已过期、人工修改、需复查等状态 |
| FR-022 | 用户可折叠解释模块 | P1 | 可折叠风险、学习提示等分区 |
| FR-023 | 用户可复制解释内容 | P1 | 可复制单行解释或整段解释 |
| FR-024 | 用户可导出解释 | P2 | 可导出当前文件解释为 Markdown |

### 4.5 持久化解释层

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-025 | 解释结果必须保存到本地 | P0 | 关闭并重新打开项目后，解释仍存在 |
| FR-026 | 解释必须与文件路径和代码位置绑定 | P0 | 重新打开同一文件能恢复解释映射 |
| FR-027 | 解释应包含生成时间和模型信息 | P1 | 可追踪解释由何时、何模型生成 |
| FR-028 | 解释应支持人工笔记 | P1 | 用户可对解释追加自己的理解 |
| FR-029 | 解释应支持状态标记 | P1 | 用户可标记未读、已读、已理解、需复查 |
| FR-030 | 解释应支持版本记录 | P2 | 可以看到历史解释版本和更新时间 |

### 4.6 行级锚定与增量更新

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-031 | 系统应检测文件是否变更 | P0 | 文件修改后能显示解释可能过期 |
| FR-032 | 未变化代码行解释应继续复用 | P0 | 仅修改某行时，其他行解释不应消失 |
| FR-033 | 系统应支持重新生成过期解释 | P0 | 用户可一键更新当前文件过期解释 |
| FR-034 | 系统应尽量处理代码行移动 | P1 | 行号变化但代码内容未变时，解释可重新定位 |
| FR-035 | 系统应基于代码块重新定位解释 | P2 | 函数内移动代码时可通过 AST 节点或代码哈希恢复映射 |

### 4.7 关系理解

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-036 | 系统应识别变量定义和使用 | P0 | 能指出某行使用的变量来自哪里 |
| FR-037 | 系统应识别函数调用关系 | P0 | 能指出当前函数调用了哪些函数 |
| FR-038 | 系统应识别导入导出关系 | P0 | 能解释 import/export 或 Python import 的来源和作用 |
| FR-039 | 系统应显示上游依赖 | P1 | 点击某行可查看其依赖的前置行或符号 |
| FR-040 | 系统应显示下游影响 | P1 | 点击某行可查看其结果被哪些行使用 |
| FR-041 | 系统应生成文件关系图 | P2 | 展示文件之间的引用关系 |
| FR-042 | 系统应生成模块级关系图 | P3 | 展示模块职责和模块间依赖 |

### 4.8 风险识别

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-043 | 系统应识别空值风险 | P0 | 例如可能访问 null/undefined/None 的字段 |
| FR-044 | 系统应识别异常处理缺失 | P0 | 例如 await 请求没有 try/catch 或错误处理 |
| FR-045 | 系统应识别安全风险 | P1 | 例如硬编码密钥、SQL 注入、未校验输入、危险 eval |
| FR-046 | 系统应识别性能风险 | P1 | 例如循环中重复请求、大文件同步阻塞 |
| FR-047 | 系统应识别依赖风险 | P1 | 例如未知包、过时 API、隐式全局变量 |
| FR-048 | 系统应聚合风险面板 | P1 | 用户可按严重程度查看文件内风险 |
| FR-049 | 风险应关联具体代码行 | P0 | 每个风险都能跳转到对应代码位置 |

### 4.9 阅读进度与认知状态

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-050 | 用户可标记单行已理解 | P1 | 行级状态可以保存和恢复 |
| FR-051 | 用户可标记函数已理解 | P1 | 函数级状态可以保存和恢复 |
| FR-052 | 系统显示文件理解进度 | P1 | 例如 42/120 行已理解 |
| FR-053 | 系统显示项目理解进度 | P2 | 例如核心文件 60% 已读 |

| FR-056 | 用户可对当前函数追问 | P1 | 回答中引用当前函数上下文 |
| FR-057 | 用户可对当前文件追问 | P2 | 回答中引用文件摘要和相关行 |
| FR-058 | 问答结果可保存为笔记 | P2 | 用户可将回答追加到解释层 |

## 5. 非功能需求

### 5.1 稳定性

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| NFR-001 | 解释保存稳定 | P0 | 重启应用后解释不丢失 |
| NFR-002 | 行级映射稳定 | P0 | 未修改代码的解释不应随机变化 |
| NFR-003 | 生成失败可恢复 | P0 | 某个文件分析失败不影响整个项目 |

### 5.2 性能

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| NFR-004 | 小文件响应速度可接受 | P0 | 500 行以内文件可在合理时间内完成解释生成 |
| NFR-005 | UI 不被分析阻塞 | P0 | 生成解释时用户仍可浏览代码 |
| NFR-006 | 支持后台队列 | P1 | 多文件分析时按队列处理 |
| NFR-007 | 增量更新优先 | P1 | 修改少量代码时不重新分析全文件 |

### 5.3 隐私与安全

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| NFR-008 | 用户明确知道代码是否会发送到模型服务 | P0 | 首次调用前弹出或展示隐私说明 |
| NFR-009 | 支持本地优先保存 | P0 | 解释层默认保存在本地项目或本地数据库 |
| NFR-010 | 支持敏感信息扫描 | P1 | 检测 API key、token、密码等敏感字符串 |
| NFR-011 | 支持模型供应商配置 | P1 | 用户可配置 OpenAI、Anthropic、本地模型等供应商 |

### 5.4 可扩展性

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| NFR-012 | 支持多语言扩展 | P1 | 架构上可以新增语言解析器 |
| NFR-013 | 支持多模型扩展 | P1 | 架构上可以替换或新增模型供应商 |
| NFR-014 | 支持解释字段扩展 | P1 | JSON schema 可以增加新字段而不破坏旧数据 |

### 5.5 可解释性

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| NFR-015 | AI 解释应结构化输出 | P0 | 输出必须符合预定义 schema |
| NFR-016 | 风险提示应说明原因 | P0 | 不能只说“有风险”，必须说明为什么 |
| NFR-017 | 解释应区分事实与推测 | P1 | 不确定内容应标记低置信度 |

## 7. 用户故事

### 7.1 逐行理解

作为一个非科班用户，我希望点击任意一行代码，就能看到这一行的解释，这样我可以逐步理解 AI 写出的项目。

验收标准：

- 点击代码行后，右侧解释面板显示该行解释。
- 解释包含表面意义、实际意义、关联意义、风险。
- 解释可以关闭后重新打开仍然存在。

### 7.2 理解函数作用

作为一个 AI 编程用户，我希望看到每个函数的职责、输入、输出和副作用，这样我能知道项目中的核心逻辑是如何组织的。

验收标准：

- 点击函数名或函数范围后，右侧显示函数解释。
- 显示参数、返回值、调用关系、风险。

### 7.3 审阅 AI 新增代码

作为一个用 AI 修改项目的人，我希望知道 AI 新增或修改的代码到底改变了什么，这样我能判断是否应该接受这些修改。

验收标准：

- 修改后的代码行被标记。
- 旧解释过期状态可见。
- 用户可以只重新生成受影响解释。

### 7.4 沉淀理解状态

作为一个学习者，我希望能标记哪些代码我已经理解了，这样我可以逐步掌握整个项目。

验收标准：

- 用户可以标记单行、函数、文件的理解状态。
- 状态可以保存和恢复。
- 文件或项目层面显示阅读进度。

## 8. 边界条件

### 8.1 超大文件

对于超过指定行数的文件，例如 2000 行以上，系统应提示用户分段分析，而不是一次性生成所有解释。

### 8.2 生成失败

当模型调用失败时：

- 保留已经生成的解释。
- 标记失败区域。
- 允许用户重试。
- 不破坏原有解释层。

### 8.3 代码与解释冲突

当代码变更导致解释不再准确时：

- 不直接删除旧解释。
- 标记为“可能过期”。
- 展示旧解释与新代码的差异。
- 允许用户重新生成。

### 8.4 AI 不确定

当模型无法确定解释时：

- 应明确标记低置信度。
- 应提示需要用户人工确认。
- 不应伪装成确定事实。

## 10. 需求核心判断

CodeReader 的 P0 需求只有一句话：

> 用户打开一个代码文件后，能够像在 IDE 中读代码一样，逐行看到稳定保存的解释，并理解这一行与上下文的关系和风险。

只要这个体验成立，CodeReader 的核心就成立。

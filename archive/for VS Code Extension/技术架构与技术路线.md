# CodeReader 技术架构与技术路线

版本：v0.1  
日期：2026-06-08  
项目名：CodeReader  
文档性质：技术方案、系统架构、实现路线说明

---

## 1. 技术目标

CodeReader 的技术目标不是简单调用 LLM 解释代码，而是构建一个稳定的“代码解释层系统”。

系统必须做到：

1. 读取真实项目代码。
2. 解析代码结构。
3. 建立行、代码块、函数、文件、模块之间的映射。
4. 调用 AI 生成结构化解释。
5. 将解释持久化保存。
6. 在代码变更后尽量复用旧解释。
7. 在 IDE 式界面中稳定呈现解释。

## 3. 推荐技术路线

### 3.1 MVP 技术路线建议

MVP 阶段建议优先选择：

> **VS Code Extension + 本地 SQLite + Tree-sitter/LSP + LLM API**

原因：

- VS Code 已经提供成熟代码编辑器、文件树、行号、语法高亮、文件监听等基础能力。
- 可以快速验证“左侧代码 + 右侧解释面板 + 稳定解释映射”的核心体验。
- 无需从零实现 IDE。
- 后续可以迁移或扩展为独立桌面应用。

### 3.2 V1 独立应用路线

当核心体验验证后，可以演进为：

> **Tauri + React + TypeScript + Monaco Editor + Local Service**

适合打造独立品牌产品 CodeReader，而不是依赖 VS Code 插件生态。

### 3.3 路线选择对比

| 路线 | 优点 | 缺点 | 适合阶段 |
|---|---|---|---|
| VS Code Extension | 启动快、文件能力强、天然 IDE 场景 | UI 自由度受限，品牌独立性弱 | MVP |
| Tauri + Monaco | 可做独立桌面产品，轻量，本地能力强 | 需要自建文件树、状态、插件能力 | V1 |
| Electron + Monaco | 生态成熟，开发快 | 包体大，资源占用高 | 可选 |
| Web SaaS | 分发简单，协作方便 | 用户代码隐私顾虑大，本地项目接入复杂 | 后期 |

## 4. 推荐技术栈

### 4.1 MVP 推荐栈

| 模块 | 推荐技术 | 说明 |
|---|---|---|
| IDE 宿主 | VS Code Extension API | 快速实现侧边栏、行标记、Webview、命令面板 |
| 前端 UI | TypeScript + React | 构建解释面板、风险面板、状态管理 |
| 代码编辑器 | VS Code 原生编辑器 | MVP 阶段不重复造编辑器 |
| 本地存储 | SQLite | 保存解释层、状态、索引、笔记 |
| 代码解析 | Tree-sitter | 多语言语法树解析、代码块定位、增量解析 |
| 语言能力 | LSP / language server | 获取定义、引用、符号、诊断等 IDE 级信息 |
| 模型调用 | Provider Adapter | 适配 OpenAI、Anthropic、本地模型等 |
| 结构化输出 | JSON Schema 校验 | 确保解释结果可存储、可渲染、可复用 |
| 文件监听 | VS Code workspace API | 监听文件变更，触发解释过期判断 |

### 4.2 V1 独立应用推荐栈

| 模块 | 推荐技术 | 说明 |
|---|---|---|
| 桌面壳 | Tauri | 用 Web 前端构建跨平台桌面应用，Rust 负责系统能力 |
| 前端 | React + TypeScript | UI 开发效率高，生态成熟 |
| 代码编辑器 | Monaco Editor | Web 端成熟代码编辑器，可模拟 VS Code 体验 |
| 本地后端 | Rust 或 Node.js sidecar | 负责文件读取、索引、模型调用、SQLite |
| 本地数据库 | SQLite | local-first 存储解释层 |
| 云端数据库 | PostgreSQL + pgvector | 后期支持云同步、语义搜索、团队空间 |
| 任务队列 | 本地队列 / BullMQ / SQLite queue | 管理解释生成任务 |
| 图谱层 | SQLite graph tables / Neo4j 可选 | MVP 不建议引入重图数据库 |

## 5. 核心技术模块

### 5.1 项目索引模块

职责：

- 扫描项目文件。
- 过滤无关目录。
- 识别语言类型。
- 生成文件哈希。
- 记录文件结构。

忽略规则：

- `.git`
- `node_modules`
- `dist`
- `build`
- `.venv`
- `__pycache__`
- 二进制文件
- 大型锁文件可选择忽略解释，但保留依赖分析

### 5.2 代码解析模块

职责：

- 将代码解析为 AST。
- 识别函数、类、变量、导入导出。
- 将行号映射到 AST 节点。
- 识别代码块边界。

解析层不要求在 MVP 中实现完整语义分析，但必须提供基础结构，避免所有解释都只依赖 LLM 猜测。

### 5.3 符号与关系模块

职责：

- 变量定义与使用关系。
- 函数调用关系。
- import/export 关系。
- 文件依赖关系。
- 当前行上游和下游关系。

MVP 中可先实现：

- 当前文件内变量定义/使用。
- 当前文件内函数调用。
- 文件 import/export。

后续再扩展跨文件语义分析。

### 5.4 AI 解释模块

AI 解释模块负责将代码结构、上下文和用户偏好转为结构化解释。

流程：

```text
代码文件
  ↓
AST / 符号 / 关系分析
  ↓
生成解释任务
  ↓
上下文组装
  ↓
调用模型
  ↓
结构化 JSON 输出
  ↓
Schema 校验
  ↓
解释质量检查
  ↓
保存到本地数据库
```

### 5.5 持久化解释层

解释层应独立于原始代码文件保存，避免污染用户代码。

推荐路径：

```text
项目根目录/
├─ src/
├─ package.json
└─ .codereader/
   ├─ codereader.sqlite
   ├─ project.json
   ├─ exports/
   └─ cache/
```

也可以提供“全局本地库”模式：

```text
用户目录/.codereader/projects/{project_id}/codereader.sqlite
```

两种模式对比：

| 模式 | 优点 | 缺点 |
|---|---|---|
| 项目内 `.codereader` | 易迁移、可随项目备份 | 可能被误提交，需要 .gitignore |
| 用户全局目录 | 不污染项目 | 项目迁移时解释不跟随 |

MVP 建议默认使用用户全局目录，并允许导出到项目内。

## 6. 行级锚定方案

行号不是稳定锚点。代码一旦插入或删除，行号会变化。因此 CodeReader 不能只用 `file_path + line_number` 保存解释。

推荐采用多重锚定：

```text
Explanation Anchor =
  file_path
  + line_range
  + code_hash
  + normalized_code_hash
  + ast_node_id
  + parent_symbol_id
  + nearby_context_hash
```

### 6.1 锚定字段

| 字段 | 说明 |
|---|---|
| file_path | 文件路径 |
| line_start / line_end | 原始行号范围 |
| code_hash | 原始代码片段哈希 |
| normalized_code_hash | 去掉空格、格式差异后的哈希 |
| ast_node_id | AST 节点标识 |
| parent_symbol_id | 所属函数/类/模块 |
| nearby_context_hash | 前后若干行上下文哈希 |

### 6.2 变更后重新定位策略

1. 精确匹配文件路径和代码哈希。
2. 若失败，匹配 normalized code hash。
3. 若失败，匹配所属函数和相似代码片段。
4. 若失败，匹配前后上下文。
5. 若仍失败，标记解释为 outdated。

### 6.3 解释状态

| 状态 | 含义 |
|---|---|
| valid | 当前解释有效 |
| stale | 代码可能变化，解释可能过期 |
| outdated | 代码已明显变化，解释不再可靠 |
| user_modified | 用户人工修改过解释 |
| low_confidence | AI 解释置信度低 |
| failed | 生成失败 |

## 8. 解释输出 Schema

AI 输出必须是结构化 JSON，不应直接将自然语言整段保存为唯一结果。

示例：

```json
{
  "target": {
    "type": "line",
    "file_path": "src/auth/login.ts",
    "line_start": 37,
    "line_end": 37,
    "code": "const user = await getUserById(userId);"
  },
  "explanation": {
    "surface_meaning": "根据 userId 异步获取用户对象。",
    "runtime_meaning": "当前异步函数会等待 getUserById 的结果，然后把结果赋值给 user。",
    "relation_meaning": "这一行依赖 userId，并为后续用户校验逻辑提供数据。",
    "global_role": "它是登录流程中获取用户实体的关键步骤。",
    "learning_hint": "需要理解 await 会暂停当前 async 函数，直到 Promise 完成。",
    "review_suggestion": "需要确认 user 为空时是否有后续处理。"
  },
  "relations": [
    {
      "type": "uses",
      "target": "userId",
      "description": "使用前面解析出的用户 ID。"
    },
    {
      "type": "defines",
      "target": "user",
      "description": "定义 user 变量供后续逻辑使用。"
    }
  ],
  "risks": [
    {
      "type": "null-check",
      "severity": "medium",
      "description": "如果 getUserById 返回 null，后续访问 user 属性可能报错。",
      "suggestion": "在后续逻辑中加入 user 不存在的判断。"
    }
  ],
  "confidence": 0.84
}
```

## 9. Prompt 设计原则

Prompt 不应只问“解释这段代码”。应提供结构化任务：

1. 当前文件整体职责。
2. 当前函数上下文。
3. 当前行或代码块。
4. 静态分析得到的符号关系。
5. 用户身份：非科班、需要解释清楚但不要过度简化。
6. 输出 schema。
7. 要求区分确定事实和推测。
8. 要求标记潜在风险。

### 9.1 行级解释 Prompt 结构

```text
你是 CodeReader 的代码解释引擎。
目标用户不是专业程序员，但需要真正掌握自己的项目。
请根据文件上下文、函数上下文、静态分析关系，对指定代码行生成结构化解释。

要求：
1. 不要泛泛解释语法，要解释这行在当前项目中的作用。
2. 如果信息不足，请标记低置信度，不要编造。
3. 必须输出 JSON，符合给定 schema。
4. 风险必须关联具体原因。
```

## 10. 解释生成策略

### 10.1 不建议逐行单独调用模型

逐行单独调用模型的问题：

- 成本高。
- 速度慢。
- 上下文不足。
- 行间解释容易割裂。

### 10.2 推荐按函数或代码块批量生成

更合理的策略：

1. 先生成文件概览。
2. 再按函数/代码块生成解释。
3. 在函数上下文中生成行级解释。
4. 将每行解释拆分保存。

这样可以保证每行解释具备上下文。

### 10.3 推荐任务粒度

| 粒度 | 用途 |
|---|---|
| 文件级 | 文件职责、导入导出、核心流程 |
| 函数级 | 函数输入输出、执行逻辑、风险 |
| 代码块级 | if/for/try 等逻辑块解释 |
| 行级 | 每行具体作用和关系 |

## 11. 风险识别策略

风险来源包括两类：

### 11.1 静态规则识别

适合确定性检测：

- 硬编码密钥。
- 使用 eval。
- SQL 拼接。
- 未处理 Promise。
- 空 catch。
- 明显未使用变量。

### 11.2 LLM 推理识别

适合语义判断：

- 业务逻辑是否可疑。
- 边界条件是否遗漏。
- AI 是否生成了冗余逻辑。
- 某个实现是否和文件职责不一致。

MVP 应结合两者，避免完全依赖 LLM。

## 14. 技术风险

### 14.1 行级锚定风险

代码频繁变化后，解释可能难以准确重定位。

应对：

- 多重锚定。
- AST 辅助。
- diff 算法。
- 明确标记过期。

### 14.2 LLM 幻觉风险

模型可能误解代码或编造意图。

应对：

- 静态分析提供事实基础。
- 输出置信度。
- 区分事实与推测。
- 用户可标记纠错。

### 14.3 成本风险

全项目逐行解释成本较高。

应对：

- 按需分析。
- 按函数批量生成。
- 缓存解释。
- 增量更新。
- 本地模型可选。

### 14.4 隐私风险

用户代码可能包含商业秘密或密钥。

应对：

- 本地优先。
- 敏感信息扫描。
- 明确模型调用提示。
- 支持用户选择模型供应商。

- Monaco Editor：Web 端代码编辑器，官方说明其为 VS Code 提供编辑器能力，并采用 MIT License。
- Tree-sitter：解析器生成工具与增量解析库，可为源码构建 concrete syntax tree，并随编辑高效更新。
- Language Server Protocol：定义编辑器/IDE 与 language server 之间的协议，用于自动补全、跳转定义、查找引用等能力。
- SQLite：适合本地持久化，官方说明其源码处于 public domain。
- Tauri：可用现有 Web 技术栈构建跨平台桌面应用，应用逻辑由 Rust 承担。
- pgvector：PostgreSQL 向量扩展，后期可用于云端语义搜索和解释层检索。

## 16. 技术核心判断

CodeReader 的技术难点不在“调用 AI 解释代码”，而在：

> **如何把 AI 解释稳定绑定到真实代码结构上，并在代码变化后继续维护这层解释。**

因此，CodeReader 的核心技术资产应是：

1. 行级锚定系统。
2. 结构化解释 schema。
3. 解释持久化数据库。
4. 代码结构与关系索引。
5. 增量更新机制。

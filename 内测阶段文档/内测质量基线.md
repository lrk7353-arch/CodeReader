# 内测质量基线

## 1. 完成定义

一项内测改动只有同时满足以下条件才算完成：

1. 行为目标和不在范围内的内容已写清楚。
2. 数据结构变化包含顺序迁移和旧库升级测试。
3. 用户可见失败不会破坏已有解释、阅读状态或模型凭据。
4. 新逻辑具有与风险相匹配的自动化测试。
5. `npm test`、`npm run build`、`npm run cargo:test`、`npm run cargo:clippy`、`npm run cargo:check` 全部通过。
6. 文档、版本和用户界面中的阶段定位一致。

## 2. 数据兼容

- SQLite 使用 `PRAGMA user_version` 作为迁移游标。
- 迁移只允许按版本顺序前进，并在事务中完成。
- 应用遇到高于自身支持范围的数据库版本时必须拒绝写入并给出明确错误。
- 发布前至少验证：全新数据库、`v0.10.0-mvp` 用户库、迁移失败回滚。

## 3. 测试分层

| 层级 | 最低覆盖 |
|---|---|
| TypeScript 单元测试 | 纯函数、状态派生、错误映射、项目导航 |
| React 容器冒烟测试 | 应用可渲染、阶段身份和主要入口存在 |
| Rust 单元测试 | 解析、上下文、变更检测、迁移、持久化 |
| Provider 测试 | 成功、超时/连接失败、无效结构、结构修复重试 |
| 发布验收 | 安装、启动、重启恢复、旧库升级、卸载后数据保留 |

## 4. 错误与诊断

- 新增错误优先使用稳定分类，例如 `llm.timeout`、`llm.connection`、`llm.invalid_response`。
- 用户消息说明可采取的动作，不直接展示无上下文的底层堆栈。
- 日志和错误不得包含 API Key、完整敏感源码或系统凭据。
- 完整跨模块 `AppError` 枚举在 Beta 2 前完成。

## 5. Git 与评审

- 分支使用 `codex/`、`feature/` 或 `fix/` 前缀，单次提交保持一个清晰目的。
- 不提交构建产物、临时目录、下载标记、用户数据库或凭据。
- PR/合并前检查 `git diff --check`、CI 结果和未跟踪文件。
- 不因整理目录而混入无关的大规模重命名。

# 发布签名与安全门禁

## 当前状态

CodeReader 的 Windows 安装包目前未进行 Authenticode 签名。`publisher` 配置只影响安装包元数据，不等于可信代码签名，也不会消除 SmartScreen 的未知发布者提示。

## 分发规则

- 未签名包只能用于开发者本人或已知内测成员，并必须明确提示“未签名内部测试版”。
- 面向不受控用户、公开下载页或自动更新渠道分发前，必须完成安装包签名、时间戳和发布后验证。
- 不以关闭 SmartScreen、指导用户忽略系统告警或使用自签名证书冒充正式发布作为替代方案。

## 正式签名前置条件

1. 确认真实发布主体和与证书一致的 Publisher 名称。
2. 获取受 Windows 信任的代码签名证书，明确普通 OV 或 EV 证书的保管方式。
3. 私钥使用硬件令牌、受控证书库或签名服务；PFX、密码和访问令牌不得提交 Git。
4. CI/发布机只授予发布任务所需的最小权限，并记录证书指纹和签名时间。
5. 使用可信 RFC 3161 时间戳服务，避免证书到期后已有签名立即失效。

## 发布验收

对最终 `.exe` 和 `.msi` 逐个验证：

```powershell
Get-AuthenticodeSignature .\artifacts\windows-x64\CodeReader*.exe
Get-AuthenticodeSignature .\artifacts\windows-x64\CodeReader*.msi
signtool verify /pa /all /v .\artifacts\windows-x64\CodeReader*.exe
signtool verify /pa /all /v .\artifacts\windows-x64\CodeReader*.msi
```

验收记录至少包含版本、SHA-256、签名主体、证书指纹、时间戳、验证结果和产物来源。任一产物未签名、签名无效、主体不匹配或哈希与清单不一致时，发布必须失败。

## Beta 4：RC 前真实验收与发布链路硬化（计划中）

Beta 4 不再继续补 MVP 能力，也不以局部功能堆叠为目标。它的定位是把 Beta 1-3 建好的工程基础拉到“可面向更广泛内测受众”的状态：真实项目能跑、失败能统计、发布链路能复验、用户体验不打断阅读，最后形成可进入 RC 候选的证据包。

- [主线 1] Windows release-chain smoke：对 `npm run release:windows` 的真实产物执行安装、启动、打开示例、打开单文件、打开项目、卸载后数据保留、升级覆盖安装、release manifest / SHA-256 / signing manifest 一致性检查。无真实证书时必须明确记录为“未签名内部测试版”，不得伪装为正式签名。
- [主线 2] 真实项目验证：准备至少 3 组真实项目样本（小型、常规中型、较大/长文件项目），记录扫描耗时、可预览文件比例、跳过原因、长结构列表体验、解释生成成功/失败原因、刷新后的解释迁移结果。验证重点是用户能不能连续完成阅读闭环，而不只是测试能不能通过。
- [主线 3] 内测退出证据：建立 Beta4 验收文档，按版本记录门禁、平台、命令、真实项目样本、已知限制和人工 smoke 结果；开始积累“连续候选版本无数据丢失 / 无升级失败”的证据。
- [主线 4] 失败原因统计与用户可见诊断：为打开项目、读取文件、生成解释、模型调用、迁移、刷新等路径补齐稳定失败分类和可记录摘要；不记录源码全文、API Key 或完整 prompt，但要能看出失败发生在哪个阶段、是否可重试、是否影响已有数据。
- [主线 5] 长结构列表与大文件体验：完善长函数/结构列表的折叠、展开、显示上限或分组策略，保证用户既能看到文件内部结构，也能回到项目结构；大文件预览、跳过、只读提示和上下文构建必须有明确边界。
- [主线 6] Linux 桌面 smoke 补全：在 Beta3 已通过 `verify:linux` 和 `tauri dev` 启动的基础上，补交互清单：窗口可见、打开文件、打开项目、模型设置、基础生成入口。Linux 仍不是正式发行承诺，但开发验证不能只停留在启动成功。
- [主线 7] Rust 长模块继续按职责拆分：优先处理 `context_builder.rs` 和 `code_service.rs` 中可独立测试的职责边界；目标是降低 review 和回归成本，不追求机械行数下降。拆分必须保持已有行为和测试稳定。
- [主线 8] 文案资源层与受众扩大：把模型设置、生成确认、生成中状态、错误提示、文件不可预览原因等高频用户文案纳入 copy resource；英文入口不要求完整商业化翻译，但必须避免新增硬编码中文扩大债务。
- [后续/外部前置] 真实 Authenticode 证书获取、`require-signed` CI 强制门禁、第二非 OpenAI provider/Ollama 协议、团队协作、云同步和完整知识图谱不作为 Beta4 阻塞项；其中签名证书是公开分发前硬前置，不是代码任务能伪造完成的事项。

Beta 4 的完成标志不是“又加了一批功能”，而是仓库里有可复验的 RC 前证据：真实项目验证记录、Windows release-chain smoke 记录、Linux 桌面交互记录、完整门禁输出、已知限制清单，以及与这些证据匹配的自动化测试。

## 内测退出标准

1. 至少两个连续候选版本没有数据丢失或无法升级问题。
2. 核心阅读闭环在真实项目中完成率和失败原因可统计。
3. 严重缺陷具有自动化回归测试。
4. 安装、升级、重启恢复和模型失败降级通过验收。
5. 已知限制、隐私边界和平台支持范围对内测用户清晰可见。
6. Windows 公开分发版本必须完成 Authenticode 签名，并在 CI 中通过 `node scripts/verify-authenticode.mjs --require-signed` 校验。
